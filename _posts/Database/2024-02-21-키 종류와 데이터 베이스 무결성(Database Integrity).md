# ☻ 키 종류와 데이터 베이스 무결성(Database Integrity)

2024년 2월 21일

# ☻ 데이터 베이스 무결성

데이터베이스 무결성이란 데이터베이스의 데이터가 정확하고 일관되며 신뢰할 수 있다는 것을 보장하는 규칙이나 제약사항을 말한다. 즉, DB에 저장된 데이터 값과 그것이 표현하는 현실 세계의 실제 값이 일치하다는 것이다.

이는 데이터의 정확도, 일관성, 유효성, 안정성을 보장하기 위해 중요하다.

## 예를 들어

가게에서 물건을 사면, 그 가게의 재고 데이터는 자동으로 업데이트되어야 한다. 만약 물건을 팔았는데도 재고 데이터가 그대로라면, 그것은 데이터베이스 무결성이 깨진 것이고, 데이터가 정확하지 않고 일관성이 없다고 할 수 있다.

---

이 데이터 베이스 무결성을 지키기 위한 제약사항이나 규칙인 무결성 규정이 있다.

## ☺︎ 데이터 베이스 무결성 규정

무결성 규정의 대상으로는 도메인, 키, 종속성 등이 있다.

### '재고는 0 이상의 정수여야 한다'

이런 규정이 있을 수 있다. 이 규정을 따르지 않고, 재고를 -1로 입력하려고 하면, DB에선 오류가 발생할 것이다.

이 무결성을 설정하기 위해 주로 제약 조건(constraints)을 사용한다. 제약 조건은 테이블에 데이터가 입력, 수정, 삭제될 때 적용되는 규칙이다.

> 지금까지 단순히 키를 설정하는 것에만 그쳤지만, 공부하고 나니 기본키, 외래키 등이 있는 이유와 그 키에 해당하는 제약 조건이 더 잘 이해가 되는 것 같다.
>

---

### 식별자

테이블의 특정 레코드를 읽고, 변경하기 위해 **레코드 끼리 구분할 수 있는 고유의 키**가 필요하다. **식별자는** 레코드를 찾거나 수정하거나 삭제하는 데 필요한 '키' 역할을한다.

### 식별자의 특징

- 고유성: 식별자의 값은 해당 테이블 내에서 고유해야 한다. 즉, 두 개의 레코드가 같은 식별자를 가질 수 없다.
- 불변성: 식별자는 시간이 지나도 변하지 않아야 한다. 식별자가 변경되면, 해당 레코드를 참조하는 다른 레코드에서 문제가 발생할 수 있다.
- Not Null: 식별자는 Null 값을 가질 수 없다. 즉, 모든 레코드는 반드시 식별자 값을 가져야 한다.

### 식별자 되기 위한 조건

1. 값이 꼭 존재해야 하며
2. 구분을 위한 고유값을 가져야 한다.

---

> 사원 테이블이 있다고 가정하면
>

사번이 식별자가 되기에 적합하다. 사번은 모든 직원들에게 부여되므로 값이 반드시 존재하며, 각 직원에게 부여되므로 고유한 값이다.


| 키 | 컬럼 | ? |
| --- | --- | --- |
| 적합한 키 | 주민등록번호, 사번 | 모든 사람이 고유한 값을 가지기에 적합하다. |
| 부적합한 키 | 몸무게, 키, 직급 | 동일한 값을 가지는 사람이 많기에 기본키가 될 수 없다. |
| 애매한 키 | 이름, 핸드폰, 이메일 | 이름 : 중복값 존재할 수 있기에 불가능하고, 핸드폰과 이메일은 고유값이긴 하나 없는 사람이 있을 수 있다. 또한 값이 바뀌는 경우가 잇다. |

# ☻ 데이터 베이스의 주요 키 유형

| 키 종류 | 설명 |
| --- | --- |
| 기본키 (Primary Key) | 각 레코드를 고유하게 식별하는 키. 테이블에서 각 레코드는 기본키에 의해 고유하게 식별될 수 있다. |
| 외래 키 (Foreign Key) | 한 테이블의 필드가 다른 테이블의 기본 키를 참조하는 것. 이는 두 테이블 간의 관계를 설정하는데 사용되며, 데이터의 일관성을 유지하는데 중요한 역할을 한다. |
| 복합키 (Composite Key) | 두 개 이상의 필드를 합친 것. 이들의 조합이 각 레코드를 고유하게 식별할 수 있어야 한다. 단일 필드만으로 레코드를 고유하게 식별하기 어려운 경우에 주로 사용된다. |
| 후보키 (Candidate Key) | 테이블의 레코드를 유일하게 식별할 수 있는 속성 또는 속성의 집합. 유일성과 최소성을 만족하는 키. 후보키 중에서 레코드를 가장 잘 대표하고, 가장 자주 사용되는 속성을 기본키로 선택한다. |
| 슈퍼키 (Super Key) | 테이블 레코드를 유일하게 식별할 수 있는 속성. 한 개 이상의 속성으로 구성될 수 있으며, 유일성은 보장하지만 최소성은 보장하지 않는다. |

> 최소성과 유일성은 무엇인가?
>
- **유일성(Unique)**: 키가 **고유**해야 한다는 것을 의미한다. 즉, 키의 값이 데이터베이스 내에서 단 한 번만 나타나야 한다. 예를 들어, 학생들의 학번은 각 학생에게만 해당되므로 유일성을 만족시킨다.
- **최소성(Minimal)**: 키가 반드시 필요한 **최소한의 필드**로만 구성되어야 한다. 예를 들어, 학생 테이블에서 이름과 생년월일이 합쳐진 키는 유일할 수 있지만, 이름만으로도 충분히 유일하다면 이름과 생년월일을 합친 키는 최소성을 만족시키지 못한다.

---

# ☺︎ Primary key (기본키)

기본키(Primary Key)는 데이터베이스에서 테이블의 각 레코드를 고유하게 식별하는 데 사용되는 키이다. 즉, 테이블에서 각 레코드는 기본키에 의해 고유하게 식별될 수 있다.

### 기본키의 특징

| 기본키 특징 | 설명 |
| --- | --- |
| 대표성 | 레코드를 상징하는 값이어야 한다. |
| 자주 참조하는 속성 | 기본키에는 기본적으로 인덱스가 생성되어 검색 효율이 좋다. |
| 가급적 짧은 속성 | 테이블 간의 연결고리가 되므로, 비교 속도가 빨라야 한다. |

### 기본키 지정 (컬럼 제약 vs 테이블 제약)

제약 선언 위치에 따라 컬럼 제약, 테이블 제약이 있다.

| 제약 | 설명                                                                                                                         |
| --- |----------------------------------------------------------------------------------------------------------------------------|
| 컬럼제약 | 특정 컬럼에만 적용되는 제약이다. 각각의 컬럼 선언 시에 정의되며, 해당 컬럼에만 적용되는 규칙을 설정한다. <br/>예를 들어, NULL 값을 허용하거나, 고유한 값만을 허용하는 등의 제약을 설정할 수 있다.      |
| 테이블제약 | 테이블 전체에 적용되는 제약이다.여러 컬럼을 관련 시키거나 테이블 간의 관계를 설정하는 제약을 설정할 수 있다. 두 컬럼의 조합이 항상 고유해야 하거나, 외래키를 사용하여 다른 테이블과의 관계를 설정하는 것 등이 있다. |

---

# ☺︎ 외래 키 (Foreign Key)

외래 키는 **한 테이블의 필드가 다른 테이블의 기본 키를 참조**하는 것을 말한다. 이는 두 테이블 간의 **관계를 설정**하는데 사용되며, 데이터의 일관성을 유지하는데 중요한 역할을 한다.

---

# ☺︎ 복합키 (COMPOSITE KEY)

복합키는 **두 개 이상의 필드를 합친 것**으로, 이들의 조합이 각 레코드를 고유하게 식별할 수 있어야 한다. 단일 필드만으로 레코드를 고유하게 식별하기 어려운 경우에 주로 사용된다.

기본키 필드가 꼭 하나여야 한다는 법은 없으며, 하나의 필드만으로 레코드를 특정하기 어려운 경우가 있는데, 이럴 때 복합키를 사용한다.

> 팀 프로젝트에서 복합키를 사용했었다. 1개의 야구 경기를 나타내기 위해서는 홈 팀, 어웨이 팀 두 개의 구단 Id를 참조키로 사용했고 이 둘을 복합키로 사용했다. 따라서 각 경기가 고유하게 식별되도록 했었다.
>

더 자세하게 예를 들자면,

1. 학교의 학생 테이블에서 각 학생은 고유한 학번을 가지나, 수강 테이블에서는 한 학생이 여러 과목을 수강할 수 있다. 또한,한 과목도 여러 학생에게 수강당할 수 있다.
2. 이 경우, 학생의 학번이나 과목 코드 만으로는 수강 테이블의 레코드를 유일하게 식별할 수 없다.
3. 따라서 **학번 ↔ 과목 코드 두 개의 필드를 조합**하여 복합키를 구성한다. 이로써 수강 레코드는 특정 학생이 특정 과목을 수강하는 것을 유일하게 표현할 수 있게 된다.

다음은 `수강` 테이블을 생성하고, 학번과 과목 코드를 복합키로 설정하는 SQL 쿼리문이다.

```sql
CREATE TABLE 수강 (
    학번 INT,
    과목코드 INT,
    성적 CHAR(1),
    PRIMARY KEY (학번, 과목코드)
);
```

이렇게 설정하면 `수강` 테이블의 각 레코드는 특정 학생이 특정 과목을 수강하는 것을 유일하게 표현할 수 있다. 예를 들어, 학번이 **123**인 학생이 과목 코드가 **456**인 과목을 수강한다면, 해당 레코드는 `(123, 456)`로 표현된다. 이 레코드는 학번만 또는 과목 코드만으로는 표현할 수 없는, 레코드를 유일하게 식별할 수 있는 정보를 담고 있다.

# ☺︎ 후보키 (Candidate Key)

후보키는 테이블의 **레코드를 유일하게 식별할 수 있는 속성 또는 속성의 집합이다.**

아래의 표는 '직원' 테이블을 예로 들어 후보키를 설명할 수 있다.

| 사번 | 이름 | 이메일 | 주민등록번호 |
| --- | --- | --- | --- |
| 001 | 김철수 | mailto:cheolsu@example.com | 900101-1234567 |
| 002 | 이영희 | mailto:younghi@example.com | 900102-1234567 |
| 003 | 박지수 | mailto:jisoo@example.com | 900103-1234567 |

위 테이블에서 '사번', '이메일', '주민등록번호' 각각은 모두 레코드를 유일하게 식별할 수 있으므로 **후보키가 될 수 있다**. 즉, 이 테이블에서는 세 개의 후보키가 존재할 수 있다.

하지만 이 중에서 DB 설계 시 레코드를 가장 잘 대표하고, 가장 자주 사용되는 속성을 기본키(Primary Key)로 선정한다.

---

# 실제 SQL 문으로 어떻게 적용하는가

### **기본 키 제약 조건(Primary Key constraint)**:

```sql
CREATE TABLE Students (
    ID INT PRIMARY KEY,
    Name VARCHAR(50),
    Age INT
);
```

위의 SQL문은 "Students"라는 테이블을 생성하며, 'ID' 칼럼을 기본 키로 지정한다.

---

### **외래 키 제약 조건(Foreign Key constraint)**:

```sql
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT,
    ProductID INT,
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);
```

위의 SQL문은 'CustomerID'를 외래 키로 지정하고, 'Customers' 테이블의 'CustomerID'를 참조하도록 설정한다.

---

### **체크 제약 조건(Check constraint)**:

```sql
CREATE TABLE Inventory (
    ProductID INT PRIMARY KEY,
    Stock INT CHECK (Stock >= 0)
);
```

위의 SQL문은 'Stock' 칼럼에 대해 체크 제약 조건을 설정하여 재고가 0 이상이어야 함을 보장한다.

---

### **NOT NULL 제약 조건**:

```sql
CREATE TABLE Employees (
    ID INT PRIMARY KEY,
    Name VARCHAR(50) NOT NULL,
    Age INT
);
```

위의 SQL문은 'Name' 칼럼에 NULL 값을 허용하지 않도록 설정한다.

---

### **UNIQUE 제약 조건**:

```sql
CREATE TABLE Employees (
    ID INT PRIMARY KEY,
    Email VARCHAR(50) UNIQUE,
    Age INT
);
```

위의 SQL문은 'Email' 칼럼에 중복된 값을 가지지 않도록 UNIQUE 제약 조건을 설정한다.

---

# ✏️ Appendix

### 슈퍼키, 후보키, 기본키의 차이

슈퍼키는 유일성은 보장하지만 최소성은 보장하지 않는다. 후보키는 최소성과 유일성을 모두 만족해야 한다. 즉, 후본키는 기본키가 될 수 있는 키이다. 그리고 기본키는 후보키 중에서 선택된 테이블의 각 레코드를 유일하게 식별하는 키이다.

따라서 **슈퍼키 > 후보키 > 기본키** 순으로 키의 범위가 좁혀진다.
