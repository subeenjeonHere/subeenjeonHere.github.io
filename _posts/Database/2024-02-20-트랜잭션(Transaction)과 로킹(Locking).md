# ☻ 트랜잭션(Transaction)과 로킹(Locking)

2024년 2월 20일

### 간략히 정리하면

트랜잭션은 데이터베이스에서 작업의 **완전성을 보장하는 단위**이다. 트랜잭션이란 여러 개의 데이터베이스 명령어들을 모아 놓은 것으로, 이들 명령어들은 모두 실행되거나 아무것도 실행되지 않아야 한다. 이러한 원칙을 트랜잭션의 '원자성'이라고 부른다.

예를 들어, 은행에서 두 계좌 간의 이체 작업은 하나의 트랜잭션으로 볼 수 있다. 이 작업은 송금자 계좌에서 돈을 빼는 작업과 수신자 계좌에 돈을 넣는 작업으로 구성되며, 이 두 작업은 모두 성공적으로 완료되어야만 한다. 만약 둘 중 하나라도 실패하면, 전체 작업은 취소되어야 한다. 이처럼 트랜잭션은 데이터의 **일관성을** 보장한다.

로킹은 동시성 제어를 위한 메커니즘이다. 여러 사용자가 **동시에 같은 데이터에 접근할 때 데이터의 일관성을 유지**하기 위해 사용된다.

예를 들어, 두 사용자가 동시에 같은 계좌에서 돈을 인출하려고 할 때, 로킹을 통해 한 사용자의 작업이 끝날 때까지 다른 사용자의 접근을 막을 수 있다. 이를 통해 계좌의 잔액이 음수가 되는 등의 문제를 방지할 수 있다.

---

# ☻ 트랜잭션(Transaction)

1. ‘쪼갤 수 없는 업무 처리의 최소 단위’를 말한다.
2. 데이터 베이스의 상태를 변환시키는 하나의 논리적 기능을 수행하기 위한 작업의 단위 또는 한꺼번에 모두 수행되어야 할 일련의 연산들을 말한다.

예를 들어, A라는 사람이 B라는 사람에게 1,000원을 지급하고 B가 그 돈을 받은 경우, 이 거래 기록은 더 이상 작게 쪼갤 수 없는 하나의 **트랜잭션을 구성한다.**

만약 A는 돈을 지불했으나 B는 돈을 받지 못했다면 그 거래는 성립되지 않는다. 이 처럼 A가 돈을 지불하는 행위와 B가 돈을 받는 행위는 **별개로 분리될 수 없으며 하나의 거래내역으로 처리**되어야 하는 단일거래이다. ⇒ 이런 최소 단위 거래를 트랜잭션이라고 한다.

```sql
예시) 사용자 A가 사용자 B에게 1,000원을 송금한다.

1. 사용자 A의 계좌에서 1,000원을 차감한다. : UPDATE를 통해 사용자 A 잔고변경
2. 사용자 B의 계좌에 1,000원을 추가한다. UPDATE를 통해 사용자 B 잔고변경

작업단위: 출금 UPDATE + 입금 UPDATE
==> 통틀어 하나의 트랜잭션이라고 한다. 두 쿼리문이 모두 성공적으로 완료되어야만 하나의 작업이 완료된다.
```

---

# 트랜잭션의 목적은?

데이터 베이스 서버에 여러 개의 클라이언트가 동시에 액세스하거나 응용 프로그램이 갱신을 처리하는 과정에서 중단될 수 있는 경우 등 데이터 부정합을 방지하고자 할 때 사용한다.

---

# 트랜잭션 특징

- 데이터 베이스 시스템에서 병행 제어 및 회복 작업 시 처리되는 작업의 논리적 단위이다.
- 사용자가 시스템에 대한 서비스 요구 시, 시스템이 응답하기 위한 상태 변환 과정의 작업 단위이다.
- 하나의 트랜잭션은 정상적으로 처리가 완료되면 커밋(Commit)되거나 오류가 발생할 경우 롤백(Rollback)된다.

---

# ☺︎ 트랜잭션 성질 (ACID 성질)

트랜잭션의 ACID 성질은  Atomicty, Consistency, Isolation, Durablity인 **원자성, 일관성, 독립성, 영속성이 있다.**

왜 이 트랜잭션 성질이 중요한가. 데이터 베이스의 안정성과 신뢰성을 보장하기 위해.

---

### Atomicity (원자성)

All or nothing

트랜잭션 연산은 데이터 베이스에 모두 반영되든지, 아니면 전혀 반영되지 않아야 한다.

트랜잭션 내 모든 명령은 반드시 완벽히 수행되어야 하며, 모두가 완벽히 수행되지 않고 어느 하나라도 오류가 발생하면 트랜잭션 전부가 취소되어야 한다.

> SavePoint
>

트랜잭션 내 중간지점으로, 트랜잭션 중간에 오류가 발생할 경우 처음부터 롤백하는 것이 아니라 **savepoint** **지점까지만 롤백**한다. 이를 통해 트랜잭션의 효율성을 높일 수 있다. 즉, 부분적으로 롤백하는 것.

### Consistency(일관성)

트랜잭션이 그 실행을 성공적으로 완료하면 언제나 **일관성있는 데이터베이스 상태로 변환**한다. 시스템이 가지고 있는 고정 요소는 트랜잭션 수행 전과 후의 상태가 같아야 한다.

---

### Isolation(독립성)

**각각의 트랜잭션은 서로에게 영향을 주지 않아야 한다.**

즉, 동시에 여러 개의 트랜잭션이 실행되더라도, **각 트랜잭션은 서로 독립적으로 실행**되어야 한다. 어떤 트랜잭션에서 오류가 발생하더라도, 다른 트랜잭션에는 영향을 주지 않아야 한다.

---

### Durability(영속성)

성공적으로 완료된 트랜잭션의 결과는 시스템이 고장나더라도 영구적으로 반영되어야 한다.

### 컴퓨터 공학에서 영속성(Persistence)란?

JPA를 공부하면서 영속성에 대해 접한적이 있다. 컴퓨터 공학에서 영속성이 어떤 의미를 갖는가?

데이터가 네트워크 오류, 시스템 장애, 프로그램 오류 등 예기치 못한 상황에도 불구하고 **영구적으로 보존되는 특성을** 의미한다.

> 오류, 시스템 장애가 있는데 데이터가 어떻게 영구보존이 가능한지 궁금하다.
>

데이터의 영속성은 트랜잭션의 '지속성(Durability)' 속성으로 보장된다고 한다. 트랜잭션이 성공적으로 완료되면, 그 결과는 영구적으로 데이터베이스에 저장되는 것이 지속성인데, 이는 데이터베이스 관리 시스템(DBMS)이 디스크 같은 영구 저장 매체를 사용하여 데이터를 저장하고, 트랜잭션의 결과를 디스크에 즉시 기록한다.

또한, 대부분의 DBMS는 로그를 사용하여 데이터의 영속성을 보장한다. 트랜잭션이 커밋되면, 해당 트랜잭션에 의해 수행된 모든 변경 사항이 로그에 기록된다. 시스템이 실패하는 경우, DBMS는 이 로그를 사용하여 마지막으로 커밋된 상태로 데이터베이스를 복구할 수 있다.

> 그럼 로그 파일이 손실되면?
>

로그 파일이 손실되면, 그 이후의 데이터 복구는 매우 어려워진다고 한다. 검색 해보니, MySQL의 경우엔  MySQL 바이너리 로그, 데이터 베이스 레플리케이션으로 복구를 진행한다.

# ☺︎ 트랜잭션 연산

- Commit, Rollback 명령어에 의해 원자성을 보장받는다.

## 커밋 연산 (Commit Operation)

커밋 연산은 **트랜잭션이 성공적으로 종료된 경우에 수행되는 연산**이다. 커밋 연산을 수행하면, 트랜잭션에 의해 수행된 모든 업데이트 연산이 데이터베이스에 반영된다. 한 번 커밋 연산이 수행되면, 이후에는 트랜잭션을 취소할 수 없다.

## 롤백 연산 (Rollback Operation)

롤백 연산은 **트랜잭션이 실패하여 종료된 경우에 수행되는 연산**이다. 롤백 연산을 수행하면, 트랜잭션에 의해 수행된 모든 업데이트 연산이 취소되고, 트랜잭션을 시작하기 이전의 상태로 복구된다. 롤백 연산 후에는 트랜잭션에 의한 모든 변경 사항이 취소되므로, 트랜잭션의 원자성을 보장한다.

---

# ☺︎ 병행 제어 (일관성 주요 기법)

### 병행제어(Concurrency Control)

병행제어는 다수 사용자 환경에서 여러 트랜잭션 연산을 수행할 때, 데이터 베이스 일관성 유지를 위해 **상호작용을 제어**하는 기법이다.

### 병행제어 목적

1. 데이터 베이스의 공유를 최대화한다.
2. 시스템의 활용도를 최대화한다.
3. 데이터 베이스의 일관성을 유지한다.
4. 사용자에 대한 응답 시간을 최소화한다.

### 병행제어 미보장 시 문제점

**갱신 손실, 현황 파악오류, 모순성, 연쇄복귀**가 있다.

| 문제점 | 설명 |
| --- | --- |
| 갱신 손실 | 두 개의 트랜잭션이 동시에 같은 데이터를 갱신할 때, 한 트랜잭션의 갱신 결과가 다른 트랜잭션에 의해 덮어쓰여지는 현상. |
| 현황 파악오류 | 한 트랜잭션의 중간 수행 결과를 다른 트랜잭션이 참조하여 발생하는 오류  |
| 모순성 | 두 개의 트랜잭션이 동시에 실행될 때 데이터의 일관성이 깨지는 현상. (일관성 결여)예를 들어, 한 트랜잭션이 A계좌에서 B계좌로 이체하는 도중, 다른 트랜잭션이 B계좌의 잔액을 조회하면, 이체 전후의 잔액이 모순을 일으킬 수 있음. |
| 연쇄복귀 | 여러 트랜잭션이 데이터를 공유하여 작업을 수행할 때, 한 트랜잭션의 실패가 다른 트랜잭션들에게 연쇄적인 실패를 유발하는 현상.예를 들어, 복수의 트랜잭션이 데이터 공유 시 특정 트랜잭션이 처리를 취소할 경우 트랜잭션이 처리한 곳의 부분을 취소하지 못하는 오류 |

---

### 병행제어 기법 종류

| 기법 | 설명 |
| --- | --- |
| 로킹(Locking) | 하나의 트랜잭션이 실행하는 동안, 특정 데이터 항목에 대해 다른 트랜잭션이 동시에 접근하지 못하도록 상호 배제 기능을 제공하는 기법 |
| 타임스탬프(Timestamp) | 트랜잭션과 트랜잭션이 읽거나 갱신한 데이터에 대해 트랜잭션이 실행을 시작하기 전에 타임 스탬프를 부여하여 부여된 시간에 따라 트랜잭션 작업을 수행하는 기법 |
| 낙관적 검증(Optimistic Validation) | 트랜잭션이 어떠한 검증도 수행하지 않고 일단 트랜잭션을 수행하고, 트랜잭션 종료 시에 검증을 수행하여 데이터 베이스에 반영하는 기법. |
| 다중 버전 동시성 제어(MVCC) | 트랜잭션의 타임 스탬프와 접근하려는 데이터의 타임 스탬프를 비교하여 직렬 가능성이 보장되는 적절한 버전을 선택하여 접근하도록 하는 기법 |

# ☺︎ 트랜잭션 상태

![image](https://github.com/subeenjeonHere/subeenjeonHere.github.io/assets/108563359/e47cccaa-496e-4885-8f70-cce067611621)


활동 → 부분완료 → 실패 → 완료 → 취소(철회)

| 상태 | 설명 |
| --- | --- |
| 활동(Active) | 트랜잭션이 실행 중인 상태. 트랜잭션의 시작 시점부터 실행 연산을 수행하는 동안의 상태 |
| 부분 완료(Partially Committed) | 트랜잭션의 마지막 연산까지 실행했지만, Commit 연산이 실행되기 전의 상태 |
| 실패(Failed) | 트랜잭션 실행에 오류가 발생하거나, 조건이 만족되지 않아 중단된 상태 |
| 완료(Committed) | 트랜잭션이 성공적으로 종료되어 Commit 연산을 실행한 후의 상태. 이 상태가 되면 트랜잭션에 의한 변경 내용은 데이터베이스에 영구적으로 반영 |
| 철회(Aborted) | 트랜잭션이 비정상적으로 종료되어 Rollback 연산을 실행한 후의 상태. 이 상태가 되면 트랜잭션 실행 이전 상태로 되돌아감 |

---

### 트랜잭션 사용 시 주의사항

작업의 단위가 크다면 그만큼 시스템에 부하를 줄 수 있으므로, 작업의 단위를 적절히 조절해야 한다. **즉, 여러 개의 트랜잭션으로 쪼개서 트랜잭션의 범위를 최소화 해야한다.**
