# 객체지향 프로그래밍이란?

객체지향(Object-Oriented)이란 컴퓨터 프로그래밍의 패러다임 중 하나이며, 프로그램을 객체들의 집합으로 보고 이들이 서로 상호작용하도록 설계하는 것이다.

이는 프로그램을 객체들의 모음으로 보는 관점을 제공하며, 이 객체들은 서로 메시지를 주고 받아 상호작용한다. 객체는 상태와 행동을 가지며, 이러한 특성을 통해 캡슐화, 상속, 다형성 등의 특징을 나타낸다.

---

### 자바는 객체지향이다

객체 지향을 공부할 것이다. 그래서 객체지향 언어이자 내 주언어 자바를 더 알아보고 싶어졌다.

자바로 뭘 할 수 있을까? 웹 프로그래밍, 안드로이드 개발(적합하진 않다.), 게임 개발, 데이터 베이스 처리 등등.

그럼 자바로 할 수 없는 것은? 시스템 프로그래밍, 운영체제, 드라이브, 커널 등 Low Level 시스템 개발에는 C, C++ 같은 언어가 더 적합하다.

또한, 높은 성능을 요구하는 프로젝트에서도 자바는 적합하지 않다. 자바는 GC(메모리를 자동 회수), JIT 컴파일러(프로그램 실행 시점에 바이트 코드를 기계어로 변환)와 같은 기능 때문에 실행 속도가 느릴 수 있다. 따라서, 고성능이 중요한 분야에선 C, C++과 같은 언어가 적합하다. C++은 컴파일 언어로 소스코드는 프로그램 실행 전에 기계어로 변환된다. 실행 전에 전체가 기계어로 컴파일 되어, 실행 시간에 추가적인 컴파일 과정이 필요하지 않다.

> 실행 전 컴파일 vs 실행 시점 컴파일 차이
>

C++처럼 실행 전에 컴파일 되는 언어들은 코드가 프로그램 실행 전에 기계어로 변환되어, 결과로 나온 바이너리 코드들은 실행 시 추가적 변환이 필요가 없다. 실행 속도를 빠르게 하지만, 모든 코드가 컴파일되어야 하므로 초기 컴파일 시간이 길어질 수 있다.

반면, JIT 컴파일러를 사용하는 자바와 같은 언어들은 코드가 프록램 실행되는 동안 기계어로 번역된다. JIT 컴파일러는 코드의 일부가 실행되어야 할 때 해당 부분을 번역하고, 동일한 코드가 재실행되어야 할 땐 이미 컴파일 된 언어를 재사용한다. 코드가 실행되는 동안에만 필요한 부분을 기계어로 번역하므로 이미 컴파일된 코드를 재사용하기 때문에 실행 속도를 높일 수 있다. 하지만 코드가 실행될 때마다 컴파일 과정이 필요하므로, 실행 속도가 느려질 수 있다.

### 객체지향 프로그래밍의 주요 구성 요소

| 구성 요소 | 설명                                                                                            |
| --- |-----------------------------------------------------------------------------------------------|
| 클래스(Class) | 객체를 생성하기 위한 템플릿이며, 객체의 상태를 나타내는 필드와 객체의 행동을 나타내는 메서드로 구성되어 있다.                                |
| 객체(Object) | 클래스에 의해 생성된 실체로, 객체는 클래스에서 정의한 것을 토대로 메모리에 할당되며 상태와 행동을 가지고 있다. 또한, 각 객체마다 상태와 식별성을 갖는다.      |
| 메서드(Method) | 클래스 내에 정의된 함수로, 객체의 행동을 나타내며, 객체의 상태를 변경하거나 객체의 상태를 활용하여 연산을 수행한다.                            |
| 인스턴스(Instance) | 객체지향 기법에서 클래스를 통해 만든 실제의 실형 객체로, 클래스에 속한 각각의 객체다. 실제로 메모리상에 할당된다.                             |
| 메시지(Message) | 객체 간 상호작용하기 위한 수단으로, 객체에 어떤 행위를 하도록 지시하는 방법이다. 객체 간의 상호작용은 메시지를 통해서 이뤄지고, 메시지는 객체 → 객체로 전달된다. |
| 속성(Property) | 한 클래스 내에 속한 객체들이 가지고 있는 데이터 값들을 단위별로 정의한 것이다.                                                 |

### 다시 한 번 정리하면, 객체란

실제 세계에서 존재하는 사물이나 개념 등을 프로그래밍에서 표현한 것이다. 사람, 물건, 장소 등 실제 세계의 개체를 소프트웨어 세계에서 '객체'라는 개념으로 표현한다. 예를 들어, '학생'이라는 객체가 있다면, 이 객체는 '이름', '나이', '학번' 등의 속성을 가질 수 있고, '공부하다', '시험보다' 등의 행동을 할 수 있다. 이처럼 객체는 속성(상태)과 메서드(행동)로 구성된다.

---

### 객체 지향의 주요 기법

| 기법 | 설명 |
| --- | --- |
| 상속(Inheritance) | 상위 클래스의 속성과 메서드를 하위 클래스가 물려받는 것을 의미. 코드 재사용성을 향상시키고, 유지 보수를 용이하게 한다. |
| 추상화(Abstraction) | 복잡한 시스템을 단순화시키는 과정. 필요한 속성과 메서드만을 가진 클래스를 정의하여 복잡도를 줄인다. |
| 캡슐화(Encapsulation) | 데이터와 그 데이터를 처리하는 메서드를 하나로 묶는 것. 객체의 내부 데이터를 외부로부터 보호한다. 또한, 객체의 세부 내용이 외부에 은폐되어 변경으로 인한 오류의 파급효과가 적다.  |
| 다형성(Polymorphism) | 하나의 메서드나 클래스가 있을 때 이것들이 다양한 방법으로 동작하는 것을 의미. 코드의 가독성을 높이고, 유지 보수를 용이하게 한다. |
| 정보 은닉(Information Hiding) | 코드 내부 데이터와 메소드를 숨기고, 공개 인터페이스를 통해서만 접근이 가능하도록 하는 코드 보안 기술이다. 필요하지 않은 정보는 접근할 수 없도록 하여, 고려되지 않은 사이드 이펙트는 최소화한다. |

---

### 다형성의 오버로딩과 오버라이딩

### 오버로딩(Overloading)

같은 이름의 메소드가 여러 개 있지만, **매개변수의 유형과 개수가 다르게하여 같은 이름의 메소드를 여러개 정의**하는 것을 의미한다. 이를 통해 같은 기능을 수행하는 메소드들을 같은 이름으로 관리할 수 있어 코드의 가독성을 높여준다.

### 오버로딩(Overloading) 예시

```java
class Overload {
    void demo(int a) {
        System.out.println("a: " + a);
    }
    void demo(int a, int b) {
        System.out.println("a and b: " + a + "," + b);
    }
}

class MainClass {
    public static void main(String args[]) {
        Overload obj = new Overload();
        obj.demo(10);
        obj.demo(10, 20);
    }
}
```

위의 예시에서는 `demo`라는 메소드의 이름은 같지만 매개변수의 개수가 다르므로 오버로딩이 적용된다.

---

### 오버라이딩(Overriding)

**상속** 관계에 있는 상위 클래스가 가지고 있는 메소드를 **하위 클래스가 재정의하여 사용**하는 것을 의미한다. 동일한 이름과 매개변수를 가진 메소드를 하위 클래스에서 다시 작성함으로써 상위 클래스의 메소드를 덮어 쓴다. 이를 통해 하위 클래스는 상위 클래스의 동작을 재정의하거나 확장할 수 있다.

### 오버라이딩(Overriding) 예시

```java
class Animal {
    void eat() {
        System.out.println("eating...");
    }
}

class Dog extends Animal {
    void eat() {
        System.out.println("eating bread...");
    }
}

class Test {
    public static void main(String args[]) {
        Dog d = new Dog();
        d.eat();
    }
}

```

위의 예시에서는 `Dog` 클래스가 `Animal` 클래스를 상속받았고, `eat` 메소드를 재정의(override)하여 "eating bread..."라고 출력하도록 변경되었다.

### 왜 오버로딩, 오버라이딩이 필요한가

코드의 재사용성, 유지보수성, 가독성을 향상시킬 수 있다.

1. **오버로딩(Overloading): 같은 이름의 메소드를 매개변수, 개수를 다르게 하여**
    - 코드의 가독성을 향상시킨다. **같은 기능을 하는 메서드들을 같은 이름**으로 관리할 수 있다.
2. **오버라이딩(Overriding): 상속관계에서, 하위 클래스가 상위 클래스의 메소드를 재정의**
    - 코드의 재사용성을 향상시킨다. 상위 클래스의 메서드를 하위 클래스에서 재사용할 수 있으므로 코드의 중복을 줄일 수 있다.
    - 유지보수성을 높인다. 상위 클래스의 메서드를 수정하면 그 메서드를 상속받는 모든 하위 클래스에서의 동작도 함께 변경되므로, 코드 수정이 필요할 때 일관성을 유지하면서도 효율적으로 코드를 관리할 수 있다.
    - 다형성을 가능하게 한다. 같은 이름의 메서드가 다른 동작을 하는 것을 가능하게 하므로, 유연하고 확장 가능한 코드를 작성할 수 있다.

### GOF(갱 오브 포) 디자인 패턴

소프트 웨어 디자인 패턴을 크게 3가지 유형으로 분류한다.

| 패턴 유형 | 설명 |
| --- | --- |
| 생성 패턴 (Creational Pattern) | 객체가 생성되는 방식을 중심으로 디자인하는 패턴. 객체 생성과 조합을 캡슐화하여 객체가 생성되거나 변경되어도 프로그램 구조에 영향을 미치지 않게 한다. |
| 구조 패턴 (Structural Pattern) | 클래스나 객체를 조합해 더 큰 구조를 만드는 패턴. 예를 들어, 서로 다른 인터페이스를 가진 2개의 객체를 묶어 단일 인터페이스를 제공하거나, 객체들을 서로 묶어 새로운 기능을 제공한다. |
| 행위 패턴 (Behavioral Pattern) | 객체나 클래스 사이의 알고리즘이나 책임 분배에 관한 패턴. 객체의 행동 패턴을 정의하고, 복잡한 흐름을 더 단순하게 설계한다. |

### 생성 패턴

객체 생성에 관련된 패턴이다. 종류는 추상 팩토리(Abstract Factory), 빌더(Builder), 프로토타입(Prototype), 팩토리 메소드(Factory Method), 싱글톤(Singletone)이 있다.

| 패턴 종류 | 기능 |
| --- | --- |
| 추상 팩토리(Abstract Factory) | 관련성 있는 여러 개의 객체를 생성하는 인터페이스를 제공하며, 각각의 팩토리에서 생성되는 객체는 상황에 따라 다르다. |
| 빌더(Builder) | 복잡한 인터페이스를 조립하여 만드는 구조로, 복합 객체를 생성하는 방법과 객체를 구현하는 방법을 분리하여 동일한 생성 절차에서 서로 다른 표현 결과를 만들 수 있다. |
| 프로토타입(Prototype) | 처음부터 일반적 원형을 만들어 놓고, 같은 형태를 가진 객체를 복제하는 방법을 제공한다. 즉, 기존 객체를 복제하여 객체를 생성한다. |
| 팩토리 메소드(Factory Method) | 상위 클래스에서 객체를 생성하는 인터페이스를 정의하고, 하위 클래스에서 인스턴스를 생성하도록 하는 방식이다. 즉, 상위 클래스에서는 인스턴스를 만드는 방법만 결정하고, 하위 클래스에서 그 데이터 생성을 책임지고 조작하는 함수들을 오버라이딩 한다. 따라서, 인터페이스와 실제 객체를 생성하는 클래스를 분리할 수 있다. |
| 싱글톤(Singleton) | 특정 클래스의 인스턴스가 하나만 만들어지고, 어디서든지 그 인스턴스에 접근할 수 있도록 한다. |

### 구조 패턴

클래스나 객체를 조합해 더 큰 구조를 만드는 패턴이다. 종류는 Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy가 있다.

| 패턴 종류 | 설명 |
| --- | --- |
| 어댑터(Adapter) | 클래스의 인터페이스를 사용자가 기대하는 다른 인터페이스로 변환하는 패턴으로, 호환성이 없는 인터페이스 때문에 함께 동작할 수 없는 클래스들이 함께 작동하도록 해준다. |
| 브릿지(Bridge) | 추상화된 클래스와 그 구현부를 분리하여 각자 독립적으로 변형할 수 있게 하는 패턴이다. |
| 컴포지트(Composite) | 객체들의 관계를 트리 구조로 구성하여 부분-전체 계층을 표현하는 패턴으로, 사용자가 단일 객체와 복합 객체를 동일하게 다루도록 한다. |
| 데코레이터(Decorator) | 기존에 구현되어 있는 클래스에 필요한 기능을 덧붙여 설계하는 패턴으로, 확장이 필요할 때 객체간의 결합을 통해 기능을 동적으로 확장할 수 있다. 상속의 대안으로 활용할 수 있다. |
| 퍼사드(Facade) | 하위 시스템에 있는 인터페이스 집합에 대한 통합된 인터페이스를 제공하는 패턴으로, 하위 시스템을 더 쉽게 사용할 수 있도록 상위 수준의 인터페이스를 정의한다. |
| 플라이웨이트(Flyweight) | 다수의 객체로 생성될 경우, 본질적 요소를 클래스화하여 공유화한다. 즉, ‘클래스 경량화’를 목적으로 한다. |
| 프록시(Proxy) | 실체 객체에 대한 대리 객체로, 실체 객체에 접근하기 이전에 필요한 행동을 취할 수 있게 만들며, 이 점을 이용해 미리 할당하지 않아도 상관없는 것들을 실제 이용할 때 할당하게 하여 메모리 용량을 아낄 수 있다. |

### 행동 패턴

클래스의 객체들이 상호작용하는 방법과 책임을 분산하는 방법을 정의하는 패턴이다. Observer, Mediator, Strategy, Command, Visitor 등이 있다.

| 패턴 종류 | 기능                                                                                           |
| --- |----------------------------------------------------------------------------------------------|
| 옵저버(Observer) | 객체의 상태 변화를 관찰하고, 변화가 있을 때마다 해당 객체에 의존하는 객체들에게 알려주는 패턴이다.                                     |
| 미디에이터(Mediator) | 객체 간의 통신을 캡슐화하여 객체들이 직접 참조하지 않도록 하는 패턴으로, 객체 간의 결합도를 낮춘다.                                    |
| 스트래티지(Strategy) | 알고리즘을 캡슐화하여 동적으로 알고리즘을 교체할 수 있게 하는 패턴이다.                                                     |
| 커맨드(Command) | 요청 자체를 객체로 만들어서 캡슐화한다. 이를 통해 클라이언트의 요청을 객체로 관리할 수 있으며, 작업의 실행을 요청한 객체와 실제 작업을 수행하는 객체를 분리한다. |
| 방문자(Visitor) | 객체 구조 내의 클래스에 새로운 기능을 추가하지 않고도 객체에 새로운 동작을 추가하는 패턴이다.                                        |
