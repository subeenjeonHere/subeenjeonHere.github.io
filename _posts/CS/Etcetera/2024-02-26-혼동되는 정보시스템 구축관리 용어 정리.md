# ☻ 혼동되는 정보 시스템 구축 관리 용어 정리

2024년 2월 24일

### 소프트웨어 생명주기 모델(SDLC) 종류에는

폭포수 모델(Waterfall Model), 프로토타입 모델(Prototype Model), 나선형 모델(Spiral Model), 반복적 모델(Iterative Model) 이 있다.

### 폭포수 모델

강물이 폭포처럼 떨어지듯 각 단계가 **순차적으로 진행되는 모델이다.**

요구분석 → 설계 → 구현 → 테스트 → 유지보수로 진행되는데, 각 **단계가 명확히 구분되어** 있고 완벽히 마무리 된 후 다음 단계를 진행할 수 있어서 유연성이 떨어진다. **Bohem**이 제시했고, **고전적 생명주기 모형으로도** 불리는 **선형 순차모델**이다.

### 프로토타입 모델

미니 버전 소프트웨어(프로토타입)을 만들어 사용자에게 보여준다. 개발자나 발주자에게 참조 모델을 제공한다. 이렇게 하면 사용자 요구사항을 빠르게 반영할 수 있지만, 프로토타입 제작에 시간과 비용이 많이 소요될 수도 있다.

### 나선형 모델

시스템 개발 시 **위험을 최소화하기 위해,** 점진적으로 완벽한 시스템으로 개발하고자 하는 모델이다.

계획 → 위험 분석 → 개발 → 고객 평가 순서로 진행된다.

### 반복적 모델

구축 대상을 나누어, 병렬적으로 개발 후 통합하거나 반복적으로 개발하여 점증완성 시키는 SDLC 모델이다.

## 소프트웨어 개발 방법론

### 애자일 방법론?

요구사항 변화에 유연하게 대응할 수 있도록 일정한 주기를 반복하며 소프트웨어를 개발한다. 고객의 요구사항을 최대한 빠르게 반영하고, 협력과 소통을 중시하는 개발의 방법론.

### XP는?

애자일 중의 한 종류로, 빠른 개발 속도와 고객 요구사항 반영을 위해 간결한 코드, 팀워크, 고객 참여를 중요시하는 방법론이다. 개발자와 고객이 밀접하게 협력하여 개발 과정에서의 변화와 적응을 적극적으로 수용하는 방법론이다.

## 하향식, 상향식

하향식 기법엔 전문가, 델파이 기법이 있다.

### 왜 하향식인가? 😐

하향식(Top-Down) 방식은 **전체적인 틀이나 구조를 먼저 설정**한 후, **그 안에서 세부적인 요소나 기능**을 구현하는 방식이다. 이와 관련해 **전문가 기법과 델파이 기법**은 프로젝트 **전체적인 구조와 규모를 먼저 파악**하고, 그에 따라 비용이나 시간을 예측하는 방식으로 하향식에 속하게 되는 것이다.

### 상향식 방식

Loc, M/M, COCOMO, Putnam, FP, 개발 단계벽 노력 기법

## 소프트웨어 비용 추정 기법 종류

### COCOMO (Constructive Cost Model)

보헴이 제안한 모델로, 프로그램 **규모에 따라 비용을 산정한다. 필요한 노력(Man-Month)**를 기반으로 산정한다. 규모에 따라 조직, 반분리, 임베디드 형으로 나눌 수 잇다.

**조직형은** 중소규모의 소포트웨어로 **5만 라인 이하**를 개발할 때, **반분리(Semi)**는 **30만 라인 이하**, **임베디드는** **30만 라인 이상**의 초대형 규모 트랜잭션 처리 시스템, 운영체제, 실시간 처리 시스템 등의 개발에 적용된다.

### LoC(Lines of Code)

프로그램 원시 코드 라인 수의 **낙관치, 중간치(기대치), 비관치를 측정하여 예측치**를 구하고, 이를 활용해 비용을 산정한다.

### 전문가 감정과 델파이

전문가 감정은 **2명 이상의 전문가**에게 비용 산정을 의뢰한다. 델파이는 주관적 판단 방지를 위해, **다수의 전문가의 의견을 종합하여** 비용을 산정한다.

### 기능점수(Function Point)

원시 코드는 **프로그램의 구현에 이용되는 언어**에 **독립적**이다. 경험을 바탕으로 **단순, 보통, 복잡한 정도에** 따라 비용을 산정한다. 프로젝트 영향도와 가중치 합을 이용해 기능 점수를 계산한다. 즉, 소프트웨어의 기능을 기반으로 비용을 산정한다.

### Putnam (푸트넘)

푸트넘 모델은 소프트웨어 **프로젝트의 비용과 일정을 예측**하기 위한 수학적 모델이다. 개발 주기 각 단계별로 요구할 **인력 분포**를 가정하는 모형이다. 푸트넘 모델을 기초로 하여 **자동화 도구인 SLIM이 개발**되었다.

**Rayleigh-Norden 곡선의** 노력 분포도를 기초로하고, 대형 프로젝트의 노력 분포 산정에 이용한다.

## 일정 관리 기법

| 모델 | 내용 |  |
| --- | --- | --- |
| 주 공정법(CPM) | 여러 작업 수행 순서가 얽혀있는 프로젝트 일정을 계산하는 기법으로, 노드와 노드간의 연결을 통해 공정 시간 계산 |  |
| PERT | 전체 작업의 상호관계를 표시하는 네트워크. 과거에 경험이 없어 소요기간 예측이 어려운 소프트웨어에서 사용한다. 노드와 간선으로 구성, 원 노드에는 작업을 화살표 간선에는 낙관치, 중간치, 비관치 표시 | 작업 예측치 = (비관치 + (4 기대치) + 낙관치 / 6 |
| 중요연쇄프로젝트관리(CCPM) | 주 공정 연쇄법으로 자원 제약사항을 고려하여 일정을 작성하는 기법 |  |
| 간트차트 (Time-Line 차트) | 일정의 시작과 끝을 막대 도표로 표시하여 전체 일정을 한눈에 볼 수 있는, 일정 관리를 위한 바(Bar) 형태의 차트. 수평 막대의 길이는 작업에 필요한 시간을 나타냄  |  |

### PERT, 간트차트가 헷갈린다. 다시 정리하면

- PERT 차트: 작업들 간의 상호 연관성을 보여주는 도구. 작업들은 노드로 표시되고, 노드들 사이의 연결선은 작업들 간의 순서나 관계를 나타낸다.
- 간트 차트: 일정 관리 도구로, **막대 도표 형태로 표시**된다. 각 막대는 **특정 작업**을 나타내며, 막대의 길이는 그 작업에 필요한 시간을 나타낸다.

# 소프트웨어 개발 표준

### 가장 헷갈리는 용어들

ISO 종류, SPICE, CMMI 등

### 우선 ISO란 무엇인가

ISO는 **국제 표준화 기구**(International Organization for Standardization)를 말한다.  이 기구는 세계 각국의 표준화 단체들이 모여서 국제 표준을 개발하고 채택하는 기구이다.

### 그럼 그 이후의 각 번호들은?

ISO 번호는 각 **표준을 구별**하기 위해 부여된 **고유번호이다**. 예를 들어, ISO 9000은 품질 관리 시스템에 대한 표준을, ISO 14000은 환경 관리 시스템에 대한 표준을 나타낸다. ISO/IEC 27001은 정보 보안 관리 시스템에 대한 표준을, ISO/IEC 20000은 IT 서비스 관리에 대한 표준을 설명한다.

### SPICE (ISO/IEC 15504)

SPICE는 소프트웨어 개발과 관리 과정의 품질을 평가하고 개선하기 위한 국제 표준이다. 즉, 프로세스 평가를 위한 국제 표준으로 프로세스의 성숙도를 측정하고, 품질 개선을 위한 지침을 제공한다.

### SPICE 프로세스 수행 능력 수준

불안정→ 수행→ 관리→ 확립→ 예측→ 최적화 단계로 나뉜다.

| 레벨 | 수준 | 설명 |
| --- | --- | --- |
| 0 | 불안정 단계 | 프로세스가 구현되지 않았거나, 프로세스가 그 목적을 달성하지 못한 단계 |
| 1 | 수행 단계 | 프로세스의 목적이 전반적으로 이뤄진 단계 |
| 2 | 관리 단계 | 정의돤 자원의 한도 내에서 그 프로세스가 작업 산출물을 인도 |
| 3 | 확립 단계 | 소프트웨어 공학 원칙에 기반하여 정의된 프로세스가 수행 |
| 4 | 예측 단계 | 프로세스가 목적 달성을 위해 통제되고, 양적인 측정을 통해 일관되게 수행 |
| 5 | 최적화 단계 | 프로세스 수행을 최적화하고, 지속적으로 업무 목적을 만족 |

### ISO 12207

ISO 12207은 소프트웨어 **생명 주기 프로세스**에 대한 국제 표준이다. 소프트웨어 개발, 운영, 유지보수, 그리고 폐기에 이르는 모든 단계를 포괄한다.

### 12207 구성으로는

기본공정, 조직공정, 지원공정으로 나뉜다.

### ISO 9126

품3질특성 및 측정 기준을 정의하고 있는 표준이다.

## CMM, CMMI

### CMM

CMM 모델의 레벨은 **초기 → 반복 → 정의 → 관리 → 최적화 단계**

### CMM과 CMMI의 차이점은?

초기, 반복, 정의, 관리, 최적화 단계로 이뤄져있다. **CMMI와 달리 반복 단계**가 있다. CMM은 프로세스 개선을 위한 구조를 제공하지만, 개별 프로세스들이 서로 어떻게 관련되어 있는지는 설명하지 않는다.

CMMI (Capability Maturity Model Integration)는 CMM을 기반으로 하지만, 여러 프로세스들이 서로 어떻게 연결되어 있는지를 보여주는 '통합' 모델이다.

### CMMI (Capability Maturity Model Integration)

CMMI는 소프트웨어 **개발 조직의 프로세스 성숙도를 평가**하고 개선하는 모델이다. 프로세스를 표준화하고, 조직의 역량을 향상시키는 데 초점을 맞춘다. 즉 CMMI는 **조직을 중심으로 성숙도를** 평가하기 위한 모델이다.

### CMMI 단계적 표현 모형의 성숙도 레벨

| 수준 | 레벨 | 설명 |
| --- | --- | --- |
| 1 | 초기화 단계 (Initial) | 정의된 프로세스가 없고, 작업자 능력에 따라 성과가 좌우되는 단계 |
| 2 | 관리 단계 (Managed) | 특정한 프로젝트 내의 프로세스가 정의되고 수행되는 단계 |
| 3 | 정의 단계 (Defined) | 조직의 표준 프로세스를 이용해 업무를 수행하는 상태 표준화, 일관된 프로세스가 존재하는 단계 |
| 4 | 정량적 관리 단계 (Quantitatively Managed) | 정량적 기법을 활용하여 핵심 프로세스를 통제하는 단계 |
| 5 | 최적화 단계 (Optimized) | 프로세스 역량 향상을 위해 신기술 도입, 프로세스 혁신 활동을 수행하는 단계 |

# 암호화 알고리즘 다시 정리

공부해도 헷갈리는 암호화 알고리즘.

암호화 알고리즘엔 양, 단방향 알고리즘이 있는데, 양방향 알고리즘 같은 경우엔 복호화를 통해서 원본 데이터로 변환할 수 있는 알고리즘이다. 이를 다시 대칭키, 비대칭키 알고리즘으로 나눌수 있다.

# 양방향

### 대칭키

**암호화 복호화에 같은 키를 사용**하는 방식이다. 암호화와 복호화 과정이 빠르다. 종류에는 DES, SEED, ARIA, AES, IDEA, LFSR이 있다.

### 비대칭키

암호호와 복호화에 다른 키를 사용하는 방식이다. 하나의 키는 공개키로 공개되고, 다른 하나는 비밀키로 보관된다. 대칭키 암호화 방식의 키 전달 문제를 해결하지만, 암호화와 복호화 과정이 대칭키 방식보다 느리다.

종류에는 RSA, 디피헬만, ECC, Elgamal이 있다. 특히 RSA는 소인수 분해하는 수학적 알고리즘을 사용한 비대칭키 암호화 방식이다.

# 단방향(일방향)

암호화 한 후에는 복호화가 불가능하다. 주로 해시함수를 사용한다.

### 해시 함수

임의의 길이를 고정된 길이 데이터로 매핑하는 함수이다.

단방향 함수는 **입력값을 특정한 규칙에 따라 변환하여 출력값을 생성하는 함수**인데, 이때 생성된 출력값으로부터 **원래의 입력값을 찾아낼 수 없는 구조**로 되어 있다. 단방향 함수는 **정보를 압**축하는 과정을 거치는데, 이 과정에서 원래의 정보가 **일부 손실**되기 때문에 **복호화가 불가능하다.**
