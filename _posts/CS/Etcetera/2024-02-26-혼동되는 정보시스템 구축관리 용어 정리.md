# ☻ 혼동되는 CS 용어와 개념 정리

# ☻ 소프트웨어 설계

## 📌 시스템 (System)

시스템이란 공통적 목적을 갖고 있는 구성 요소들을 상호 결합해놓은 유기적 집합체이다.

### 시스템의 구성 요소로는

**입력, 출력, 처리, 제어, 피드백**이 있다.

---

## 📌 요구사항 개발 프로세스

도출 → 분석 → 명세 → 확인

### 정형 명세와 비정형 명세 기법

**정형 명세**는 **수학적 기호와 형식**을 사용하여 요구사항을 표현하는 방법.

이 방법은 요구사항이 명확하고 일관성 있게 표현되도록 하며, 모호성과 오해를 최소화한다.

> 예시로는 **Z 명세, VDM, Petri netm CSP** 등이 있다.
>

반면에, **비정형 명세**는 일반적인 언어, 다이어그램, 그래픽 등 **자연어를** 사용하여 요구사항을 표현하는 방법.

이 방법은 이해하기 쉽고, 비개발자나 비전문가와의 커뮤니케이션에 유용하나, 모호성과 오해의 여지가 있을 수 있다.

> 예시로는 **FSM, ER모델링, SADT, 사용자 스토리, 유스케이스 다이어그램** 등이 있다.
>

## 📌 유스 케이스 (Use Case)

유스 케이스 다이어그램은 개발될 시스템을 **사용자 관점에서 표현한 것이다. 사용자의 요구사항을 분석**하기 위한 도구로 사용하며, 시스템의 범위를 파악 가능하다.

### 유스 케이스 다이어그램 구성 요소

| 구성요소 | 내용 | 상세 |
| --- | --- | --- |
| 시스템 | 내부의 유스케이스들을 사각형으로 묶어 시스템의 범위를 표현한 것 |  |
| 액터 | 시스템과 상호작용을 하는 모든 외부요소, 주로 사람이나 외부 시스템을 의미하며 주액터와 부액터가 있다. | 주액터는 주로 사람이 해당하며, 왼쪽에 배치한다. 예를 들면 회원, 비회원, 고객 등이다. 부액터는 주액터의 목적 달성을 위해 시스템에 서비스를 제공하는 외부 시스템으로 조직이나 기관 등이 될 수 있다. 예를 들면 재고, 결제, 배송 시스템 등 |
| 유스케이스 | 사용자가 보는 관점에서 시스템이 → 액터 에게 제공하는 서비스나 기능을 표현한 것 |  |
| 관계 | 액터↔ 유스케이스, 유스케이스 ↔ 유스케이스 사이에서 나타남 | 확장, 포함, 일반화 |

### 관계

액터와 유스케이스, 유스케이스 유스케이스 사이에서 나타낼 수 있다고 한다. 관계의 종류로는 **확장, 포함, 일반화 관계**가 있다.

> 상품 주문 시스템에서 회원이라는 액터가 상품 주문 유스케이스와 연관이 있다면? ⇒ 회원은 상품 주문이라는 목적 달성을 위해 상품 주문 시스템과 상호작용한다. 라고 해석할 수 있다.
>

### Use Case를 정리하면

**사용자 관점**에서 표현한 다이어 그램으로, 사용자의 요구 분석을 위한 도구이고 **표현 방법**은 아래와 같다.

| 부액터 | 실선 | <<Actor>> |
| --- | --- | --- |
| 포함관계 | 점선 | <<include>> |
| 확장관계 | 점선 | <<extends>> |
| 일반화관계 | 빈 삼각형 + 실선 | 없음 |

## 📌 객체 지향 (Object-Oriented Programming)

### 객체 지향 프로그래밍의 주요 구성 요소

| 구성 요소 | 설명                                                                                            |
| --- |-----------------------------------------------------------------------------------------------|
| 클래스(Class) | 객체를 생성하기 위한 템플릿이며, 객체의 상태를 나타내는 필드와 객체의 행동을 나타내는 메서드로 구성되어 있다.                                |
| 객체(Object) | 클래스에 의해 생성된 실체로, 객체는 클래스에서 정의한 것을 토대로 메모리에 할당되며 상태와 행동을 가지고 있다. 또한, 각 객체마다 상태와 식별성을 갖는다.      |
| 메서드(Method) | 클래스 내에 정의된 함수로, 객체의 행동을 나타내며, 객체의 상태를 변경하거나 객체의 상태를 활용하여 연산을 수행한다.                            |
| 인스턴스(Instance) | 객체지향 기법에서 클래스를 통해 만든 실제의 실형 객체로, 클래스에 속한 각각의 객체다. 실제로 메모리 상에 할당된다.                            |
| 메시지(Message) | 객체 간 상호작용하기 위한 수단으로, 객체에 어떤 행위를 하도록 지시하는 방법이다. 객체 간의 상호작용은 메시지를 통해서 이뤄지고, 메시지는 객체 → 객체로 전달된다. |
| 속성(Property) | 한 클래스 내에 속한 객체들이 가지고 있는 데이터 값들을 단위별로 정의한 것이다.                                                 |

> 자바를 하는 내가 이걸 모르면 되나 싶은 객체 지향의 주요 기법들
>

### 객체 지향의 주요 기법

| 기법 | 설명 |
| --- | --- |
| 상속(Inheritance) | 상위 클래스의 속성과 메서드를 하위 클래스가 물려받는 것을 의미. 코드 재사용성을 향상시키고, 유지 보수를 용이하게 한다. |
| 추상화(Abstraction) | 복잡한 시스템을 단순화시키는 과정. 필요한 속성과 메서드만을 가진 클래스를 정의하여 복잡도를 줄인다. |
| 캡슐화(Encapsulation) | 데이터와 그 데이터를 처리하는 메서드를 하나로 묶는 것. 객체의 내부 데이터를 외부로부터 보호한다. 또한, 객체의 세부 내용이 외부에 은폐되어 변경으로 인한 오류의 파급효과가 적다.  |
| 다형성(Polymorphism) | 하나의 메서드나 클래스가 있을 때 이것들이 다양한 방법으로 동작하는 것을 의미. 코드의 가독성을 높이고, 유지 보수를 용이하게 한다. |
| 정보 은닉(Information Hiding) | 코드 내부 데이터와 메소드를 숨기고, 공개 인터페이스를 통해서만 접근이 가능하도록 하는 코드 보안 기술이다. 필요하지 않은 정보는 접근할 수 없도록 하여, 고려되지 않은 사이드 이펙트는 최소화한다. |

### 다형성과 오버로딩 그리고 오버라이딩

**오버로딩**은 **같은 이름의 메서드를 여러 개 가지면서 매개변수의 유형과 개수를 다르게** 하는 것, 반면에 **오버라이딩**은 **하위 클래스가 상위 클래스의 메서드를 재정의하는** **것**을 의미한다.

### 오버로딩, 오버라이딩이 왜 다형성이랑 관련이 있나?

다형성이란 **하나의 메서드나 클래스가 있을 때, 이것들이 다양한 방법으로 동작하는 것이다**. **클래스나 메서드는 같은 이름을 가지지만, 입력에 따라 다르게 동작**할 수 있다. 이것이 다형성의 핵심 개념이며, 이는 코드의 재사용성을 높이고 가독성을 개선하는 데 도움이 된다.

같은 이름의 메소드를 다양한 방식으로 동작할 수 있게 하므로. 특히나 재사용성 강화 → 코드의 중복 감소로 이어지는 것이라 판단했다.

### 📌 리스코프 치환 원칙

리스코프 치환 원칙은 **객체지향 프로그래밍의 중요한 원칙** 중 하나로, 이 원칙은 **서브 타입(하위 클래스)은 어디에서나 기반 타입(상위 클래스)로 교체할 수 있어야 함**을 의미한다.

### 무슨 소리인가

> 예를 들어, '동물'이라는 기반 타입이 있고 '고양이'와 '강아지'라는 서브 타입이 있다면, 어디에서나 '동물' 대신 '고양이'나 '강아지'를 사용할 수 있어야 한다는 원칙이다.
>

## 📌 HIPO (**Hierarchy Input Process Output)**

큰 시스템을 작은 부분으로 세분화하고, 그 관계를 도식적으로 나타내어 복잡한 시스템을 이해하고 분석하는 데 도움을 주는 도구로, **하향식 소프트웨어 개발을 위한 문서화 도구이다.**

또한 **기능과 자료의 의존 관계를 동시에 표현할 수 있으며, 보기 쉽고 이해하기 쉽다.**

### 왜?

하향식 개발과 HIPO가 무슨 상관인가? 왜냐면 **하향식 개발은 전체  시스템을 먼저** 보고 그것을 세부 요소로 분해하는 방식을 사용한다. 이러한 접근 방식에서 **HIPO는 계층적 복잡한 시스템을 세분화**하고, 세부 요소들의 작용을 **도식적으로 표현**해주기 때문이다.

### HIPO 차트에도 종류가 있는데

**가시적 도표, 총체적 도표, 세부적 도표**가 있다.

TRW 사에서 우주 국방 시스템 그룹에 의해, 요구사항을 명확히 기술할 목적으로 개발한 것 → RSL, REVS 사용

# ☻ 정보 시스템 구축 관리

2024년 2월 24일

### 소프트웨어 생명주기 모델(SDLC) 종류에는

폭포수 모델(Waterfall Model), 프로토타입 모델(Prototype Model), 나선형 모델(Spiral Model), 반복적 모델(Iterative Model) 이 있다.

### 폭포수 모델

강물이 폭포처럼 떨어지듯 각 단계가 **순차적으로 진행되는 모델이다.**

요구분석 → 설계 → 구현 → 테스트 → 유지보수로 진행되는데, 각 **단계가 명확히 구분되어** 있고 완벽히 마무리 된 후 다음 단계를 진행할 수 있어서 유연성이 떨어진다. **Bohem**이 제시했고, **고전적 생명주기 모형으로도** 불리는 **선형 순차모델**이다.

### 프로토타입 모델

미니 버전 소프트웨어(프로토타입)을 만들어 사용자에게 보여준다. 개발자나 발주자에게 참조 모델을 제공한다. 이렇게 하면 사용자 요구사항을 빠르게 반영할 수 있지만, 프로토타입 제작에 시간과 비용이 많이 소요될 수도 있다.

### 나선형 모델

시스템 개발 시 **위험을 최소화하기 위해,** 점진적으로 완벽한 시스템으로 개발하고자 하는 모델이다. 즉 위헙 관리에 초점을 두고 있다.

순서는 **계획 → 위험 분석 → 개발 → 고객 평가** 순서로 진행된다.

### 반복적 모델

구축 대상을 나누어, 병렬적으로 개발 후 통합하거나 반복적으로 개발하여 점증완성 시키는 SDLC 모델이다.

## 소프트웨어 개발 방법론

### 애자일 방법론?

요구사항 변화에 유연하게 대응할 수 있도록 일정한 주기를 반복하며 소프트웨어를 개발한다. 고객의 요구사항을 최대한 빠르게 반영하고, 협력과 소통을 중시하는 개발의 방법론.

### XP

애자일 중의 한 종류로, 빠른 개발 속도와 고객 요구사항 반영을 위해 간결한 코드, 팀워크, 고객 참여를 중요시하는 방법론이다. 개발자와 고객이 밀접하게 협력하여 개발 과정에서의 변화와 적응을 적극적으로 수용하는 방법론이다.

### XP의 핵심 가치

용기, 단순성, 의사소통, 피드백, 존중이 있다.

### 스크럼(Scrum)

짧은 시간의 개발을 하는 팀을 위한 프로젝트 관리 중심의 개발 방법론이다.

### 스크럼의 주요 진행과정은

제품 백로그 -> 스프린트 -> 스프린트 계획회의 -> 스프린트 백로그 -> 일일 스크럼 회의 -> 실행 가능한 제품 개발

### 스크럼 용어에는?


| 용어 | 설명 |
| --- | --- |
| 제품 백로그 | 개발한 제품에 대한 요구사항 목록 |
| 스프린트 | 반복적 개발 주기, 2-4주 정도의 짧은 개발기간으로 개발 품질을 향상화 |
| 일일 스크럼 회의 | 매일 15분정도 진행하는 스크럼 회의 |
| 스크럼 마스터 | 스크럼 프로젝트의 리더 |
| 스프린트 회고 | 스프린트 주기를 되돌아보며, 확인하고 개선하는 것 |
| 번 다운 차트 | 남아있는 백로그 시간대비 그래프로 표현한 차트 |

### 린 (Lean)

도요타의 시스템 품질 기법을 소프트웨어 개발에 적용한 것, 낭비 요소를 제거하여 품질을 향상한다. JIT, 칸반보드를 사용한다.

### 린의 7가지 원칙

낭비제거, 품질 내재화, 지식 창출, 늦은 확정, 빠른 인도, 사람 존중, 전체 최적화

## 하향식, 상향식

하향식 기법엔 전문가, 델파이 기법이 있다.

### 왜 하향식인가?

하향식(Top-Down) 방식은 **전체적인 틀이나 구조를 먼저 설정**한 후, **그 안에서 세부적인 요소나 기능**을 구현하는 방식이다. 이와 관련해 **전문가 기법과 델파이 기법**은 프로젝트 **전체적인 구조와 규모를 먼저 파악**하고, 그에 따라 비용이나 시간을 예측하는 방식으로 하향식에 속하게 되는 것이다.

### 상향식 방식

Loc, M/M, COCOMO, Putnam, FP, 개발 단계벽 노력 기법

## 소프트웨어 비용 추정 기법 종류

### COCOMO (Constructive Cost Model)

**보헴이** 제안한 모델로, 프로그램 **규모에 따라 비용을 산정한다. 필요한 노력(Man-Month)**를 기반으로 산정한다. 규모에 따라 조직, 반분리, 임베디드 형으로 나눌 수 있다.

**조직형은** 중소규모의 소프트웨어로 **5만 라인 이하**를 개발할 때, **반분리(Semi)**는 **30만 라인 이하**, **임베디드는** **30만 라인 이상**의 초대형 규모 트랜잭션 처리 시스템, 운영체제, 실시간 처리 시스템 등의 개발에 적용된다.

### LoC(Lines of Code)

프로그램 원시 코드 라인 수의 **낙관치, 중간치(기대치), 비관치를 측정하여 예측치**를 구하고, 이를 활용해 비용을 산정한다.

### 전문가 감정과 델파이 기법

전문가 감정은 **2명 이상의 전문가**에게 비용 산정을 의뢰한다. 델파이는 주관적 판단 방지를 위해, **다수의 전문가의 의견을 종합하여** 비용을 산정한다.

### 기능점수(Function Point)

원시 코드는 **프로그램의 구현에 이용되는 언어**에 **독립적**이다. 경험을 바탕으로 **단순, 보통, 복잡한 정도에** 따라 비용을 산정한다. 프로젝트 영향도와 가중치 합을 이용해 기능 점수를 계산한다. 즉, 소프트웨어의 기능을 기반으로 비용을 산정한다.

### Putnam (푸트넘)

푸트넘 모델은 소프트웨어 **프로젝트의 비용과 일정을 예측**하기 위한 수학적 모델이다. 개발 주기 각 단계별로 요구할 **인력 분포**를 가정하는 모형이다. 푸트넘 모델을 기초로 하여 **자동화 도구인 SLIM이 개발**되었다.

**Rayleigh-Norden 곡선의** 노력 분포도를 기초로하고, 대형 프로젝트의 노력 분포 산정에 이용한다.

# 📌 소프트웨어 개발 표준

### 가장 헷갈리는 용어들

ISO 종류, SPICE, CMMI 등

### 우선 ISO란 무엇인가

ISO는 **국제 표준화 기구**(International Organization for Standardization)를 말한다.  이 기구는 세계 각국의 표준화 단체들이 모여서 국제 표준을 개발하고 채택하는 기구이다.

### 그럼 그 이후의 각 번호들은?

ISO 번호는 각 **표준을 구별**하기 위해 부여된 **고유번호이다**. 예를 들어, ISO 9000은 품질 관리 시스템에 대한 표준을, ISO 14000은 환경 관리 시스템에 대한 표준을 나타낸다. ISO/IEC 27001은 정보 보안 관리 시스템에 대한 표준을, ISO/IEC 20000은 IT 서비스 관리에 대한 표준을 설명한다.

### SPICE (ISO/IEC 15504)

SPICE는 소프트웨어 개발과 관리 과정의 품질을 평가하고 개선하기 위한 국제 표준이다. 즉, 프로세스 평가를 위한 국제 표준으로 프로세스의 성숙도를 측정하고, 품질 개선을 위한 지침을 제공한다.

### SPICE 프로세스 수행 능력 수준

불안정→ 수행→ 관리→ 확립→ 예측→ 최적화 단계로 나뉜다.

| 레벨 | 수준 | 설명 |
| --- | --- | --- |
| 0 | 불안정 단계 | 프로세스가 구현되지 않았거나, 프로세스가 그 목적을 달성하지 못한 단계 |
| 1 | 수행 단계 | 프로세스의 목적이 전반적으로 이뤄진 단계 |
| 2 | 관리 단계 | 정의돤 자원의 한도 내에서 그 프로세스가 작업 산출물을 인도 |
| 3 | 확립 단계 | 소프트웨어 공학 원칙에 기반하여 정의된 프로세스가 수행 |
| 4 | 예측 단계 | 프로세스가 목적 달성을 위해 통제되고, 양적인 측정을 통해 일관되게 수행 |
| 5 | 최적화 단계 | 프로세스 수행을 최적화하고, 지속적으로 업무 목적을 만족 |

### ISO 12207

ISO 12207은 소프트웨어 **생명 주기 프로세스**에 대한 국제 표준이다. 소프트웨어 개발, 운영, 유지보수, 그리고 폐기에 이르는 모든 단계를 포괄한다.

### 12207 구성으로는

기본공정, 조직공정, 지원공정으로 나뉜다.

### ISO 9126

품3질특성 및 측정 기준을 정의하고 있는 표준이다.

## 📌 CMM, CMMI

### CMM(Capability Maturity Model)

소프트웨어 **개발 조직의 역량 수준**을 평가하기 위한 모델이다.

5단계의 성숙도 수준을 정의하는데, **초기 → 반복 → 정의 → 관리 → 최적화 단계로 나뉜다.**

### CMMI (Capability Maturity Model Integration)

CMM의 개념을 확장하여, 소프트웨어 **개발 조직의 프로세스 성숙도를 평가**하고 개선하는 모델이다.

프로세스를 표준화하고, 조직의 역량을 향상시키는 데 초점을 맞춘다. 즉 CMMI는 **조직을 중심으로 성숙도를** 평가하기 위한 모델이다.

### CMMI 단계적 표현 모형의 성숙도 레벨

| 수준 | 레벨 | 설명 |
| --- | --- | --- |
| 1 | 초기화 단계 (Initial) | 정의된 프로세스가 없고, 작업자 능력에 따라 성과가 좌우되는 단계 |
| 2 | 관리 단계 (Managed) | 특정한 프로젝트 내의 프로세스가 정의되고 수행되는 단계 |
| 3 | 정의 단계 (Defined) | 조직의 표준 프로세스를 이용해 업무를 수행하는 상태 표준화, 일관된 프로세스가 존재하는 단계 |
| 4 | 정량적 관리 단계 (Quantitatively Managed) | 정량적 기법을 활용하여 핵심 프로세스를 통제하는 단계 |
| 5 | 최적화 단계 (Optimized) | 프로세스 역량 향상을 위해 신기술 도입, 프로세스 혁신 활동을 수행하는 단계 |

### ✏️ CMM과 CMMI의 차이점은?

CMM은 초기, 반복, 정의, 관리, 최적화 단계로 이뤄져 있어, **CMMI과 달리 반복 단계**가 있다. CMM은 프로세스 개선을 위한 구조를 제공하지만, 개별 프로세스들이 서로 어떻게 관련되어 있는지는 설명하지 않는다.

반면, **CMMI** (Capability Maturity Model Integration)는 CMM을 기반으로 하지만, 여러 프로세스들이 서로 어떻게 연결되어 있는지를 보여주는 '통합' 모델이다.

# 📌 암호화 알고리즘

> 아직 헷갈리는 암호화 알고리즘
>

암호화 알고리즘엔 **양, 단방향** 알고리즘이 있는데, **양방향 알고리즘**은 **복호화를 통해서 원본 데이터로 변환**할 수 있는 알고리즘이다. 이를 다시 **대칭키, 비대칭키** 알고리즘으로 나눌수 있다.

# 양방향 - 대칭키, 비대칭키

## 대칭키

**암호화 복호화에 같은 키를 사용**하는 방식이다. 대칭키는 **암호화와 복호화 과정이 빠르다.**

종류에는 DES, SEED, ARIA, AES, IDEA, LFSR이 있다.

### 대칭키 알고리즘 종류

**Stream**, **Block** 방식으로 나뉜다.

### 1. Stream 방식

평문과 키로 데이터를 암호화 하는 방식으로, 대표 암호 알고리즘은 **RC4**

### 2. Block 방식

평문을 **블록 단위로 분할**하여 암호화한 것으로, 암호화와 복호화 연산 시에 블록의 크기에 따라 속도 차이가 발생할 수 있다.

대표 블록 암호알고리즘 방식으론 **DES**가 있다.

### Block방식 : DES

64비트의 블록 단위로 암호화하며, 56비트의 대칭키를 사용한다.

보안성이 높다는 장점이 있지만, 대칭키 암호화 알고리즘 중에선 비교적 낮은 수준의 키 길이와 블록의 크기로 인해서, 보안성이 약점이 될 수가 있다. 이를 보완하기 위해 나온 알고리즘이 **3DES**라는 알고리즘이 나왔다.

### DES → 3DES

DES를 세 번 반복하는 방식으로, 키 길이를 168 비트로 확장하여 보안성을 높이는 방법이다.

### Block방식 두 번째: AES

128, 192, 256 비트의 키를 사용하여 평문을 암호화하며, 블록 암호화 방식을 사용한다. 각 블록은 상태 행렬로 변환되어 암호화 되는데, 이 과정에서 **행렬연산**을 사용한다.

### ✏️ 대칭키 알고리즘을 다시 정리하면

대칭키 알고리즘은 **암호화와 복호화에 같은 키를** 사용하는 방식으로, Stream, Block방식으로 다시 나눌수가 있다. **Stream** 방식은 **평문과 키로 데이터를 암호화 하는 방식으로, RC4**가 대표적이다.

**Block** 방식은 평문을 **블록 단위로 암호화** 한 것이다. **DES**가 대표적인 알고리즘인데, 다른 대칭키 알고리즘에 비해서 보안적 측면에서 약점이 있다. 그래서 이를 보완하기 위해서 DES를 세 번 반복하는 **3DES** 알고리즘이 등장했다. 또한 **AES** 알고리즘이 있는데 128, 최대 256 비트의 키를 사용하여 평문을 암호화한다. **행렬 연산을 사용한다.**

대칭키 알고리즘의 종류로는 **ARIA, DES, SEED, AES, IDEA, LFSR** 방식이 있다.

## 비대칭키

**암호호와 복호화에 다른 키**를 사용하는 방식으로, 인증과 암호화를 동시에 수행할 수 있다.

하나의 키는 공개키로 공개되고, 다른 하나는 비밀키로 보관된다. 대칭키 암호화 방식의 키 전달 문제를 해결하지만, 암호화와 복호화 과정이 대칭키 방식보다 느리다.

### ✏️ 암 - 복호화에 다른 키를 사용한다는 것의 의미는 뭔가

두 개의 키가 사용되는데, 하나는 **공개키(public key)로서 누구나 알 수** 있으며, 다른 하나는 **비밀키(private key)로서 특정한 사람만**이 알 수 있다.

예를 들어, 메시지를 보낼 때 **공개키를 사용하여 메시지를 암호화하면, 이 암호화된 메시지는 비밀키를 가진 사람만이 복호화하여** 원래의 메시지를 읽을 수 있다. 보안성이 높은 대신, 대칭키 암호화에 비해 **속도가 느리고 키의 관리와 분배가 복잡**하다.

### 비대칭키 알고리즘

종류에는 **RSA, 디피-헬만, ECC, Elgamal**이 있다.

### RSA

**소인수 분해 문제의 난해성**을 기반으로 한다. 1997년 MIT 교수진에 의해 만들어졌으며 비대칭키 알고리즘임에 공개키와 비밀키를 사용해 암, 복호화를 수행한다.

### ✏️ 비대칭키 알고리즘을 다시 정리하면

비대칭키 알고리즘은 암 ↔ 복호화 시 다른 키를 사용하는 방식이다. 공개키로 공개되고, 비밀키로 특정한 사람만이 보관하거나. 따라서 공개키를 사용해 암호화를 진행하면 → 비공개키를 가진 특정 사람만이 비밀키로 복호화하여 메시지를 읽어낼 수 있다. **암 ↔ 복호화 시 같은 키를 사용하는 대칭키 알고리즘**과 달리, 두 개의 키를 사용하기에 속도가 느리고 키의 관리가 복잡하다. 하지만 보안성이 높다는 장점이 존재한다.

# 단방향(일방향)

암호화 한 후에는 복호화가 불가능하다. 주로 해시함수를 사용한다.

### 해시 함수

임의의 길이를 **고정된 길이 데이터로 매핑**하는 함수이다.

단방향 함수는 **입력값을 특정한 규칙에 따라 변환하여 출력값을 생성하는 함수**인데, 이때 생성된 출력값으로부터 **원래의 입력값을 찾아낼 수 없는 구조**로 되어 있다. 단방향 함수는 **정보를 압**축하는 과정을 거치는데, 이 과정에서 원래의 정보가 **일부 손실**되기 때문에 **복호화가 불가능**하다.

## 📌 공격 기법

## Dos(Denial of Service attack)

표적이 되는 **서버의 자원을 고갈시킬 목적으로, 대량의 데이터를 한 곳에 집중적으로 전송하는** 공격기법

## DDoS(Distributed Dos)

여러 곳에 분산된 컴퓨터에 악성코드 좀비를 퍼뜨린 뒤, **감염된 컴퓨터들 지점에서 한 곳의 서버에 대해 분산 서비스 공격을** 수행한다.

### ✏️ DoS와 DDoS의 차이점은 무엇인가?

Dos와 DDoS 공격은 모두 서버의 자원을 고갈시키는 공격 방식인데, 그 방법이 다르다.

Dos 공격은 단일한 출처에서 **대량의 데이터를 한 곳에 집중적으로 전송하는** 방식이다.

반면, DDoS 공격은 **여러 곳에 분산된 컴퓨터**를 이용해 **악성 코드를 전파**하고, 이 **감염된 컴퓨터**들(좀비 컴퓨터)을 통해 **한 곳의 서버에 대해 분산 서비스 공격**을 수행하는 방식이다. 이렇게 함으로써 공격의 흔적을 숨기고, 방어가 더 어렵게 만든다고 한다.

즉, **DoS 공격은 시스템 대 시스템 공격인 반면 DDoS 공격은 여러 개의 시스템이 하나의 시스템을 공격하는 것이다.**

## DoS 공격의 종류

### SYN 플러딩

TCP 프로토콜의 구조적 문제를 이용한 공격으로, 가용 사용자 수를 SYN 패킷만 보내 다른 사용자가 서버를 사용 불가능하게 하는 공격이다. 공격자는 ACK를 발송하지 않고, 계속 새로운 연결 요청을 하게되어 서버는 자원 할당을 해지하지 않고, 자원만 소비하여  자원이 고갈된다.

즉, **존재하지 않는 클라이언트가 접속 가능 공간에 접속한 것처럼 속여** 다른 사용자가 서비스를 이용하지 못 하게 하는 것이다.

### UDP 플러딩

대량의 **UDP 패킷을 만들어 임의의 포트 번호로 전송하여, 응답 메세지를 생성하게 하여 지속해서 자원을 고갈시키는 공격이다.**

### 스머프, 스머핑 (Smurfing)

**출발지 주소 = 공격 대상의 IP로 설정**하여, 네트워크 전체에 ICMP Echo 패킷을 직접 브로드 캐스팅하여 마비시키는 공격이다.

### ICMP(Internet Control Message Protocol)

인터넷 프로토콜(IP)의 주요 부분으로 **네트워크 컴퓨터에 문제가 있을 때 오류 메시지**를 보낸다.

### 브로드 캐스팅

네트워크에서 모든 장치나 사용자에게 동시에 메시지를 전송하는 방법

### ✏️ 쉽게 설명하면..

> 이해가 잘 가지않아, 구글링해서 번역한 것
>
- **TCP SYN 플러딩은** 마치 카페에서 모든 자리를 차지하기 위해 고객이 주문만 하고 실제로는 자리에 앉지 않는 것 → SYN 패킷만 보내 다른 사용자가 서버를 사용 불가능 하도록
- **UDP 플러딩은** 우편함에 무수히 많은 편지를 쑤셔 넣어 우편함이 터져 버리도록 만드는 것 → 대량의 UDP 패킷을 만들어 임의의 포트번호로 전송
- **스머프 공격은** 어떤 사람이 다른 사람의 이름으로 모든 사람에게 메시지를 보내서 그 사람이 메시지의 물량에 잠식되게 하는 것. 이렇게 되면 그 사람은 너무 많은 메시지로 인해 다른 일을 처리할 수 없게 된다.

### 죽음의 핑 (PoD; Ping of Death)

1. **ICMP 패킷**을 정상적 크기보다 **아주 크게 만들어 전송하면** 다수의 IP **단편화가** 발생
2. 수신 측에서는 단편화된 패킷을 **재조립하는 과정에서 많은 부하가 발생**
3. **재조합 버퍼의 오버 플로우**가 발생하여 정상적인 서비스를 하지 못 함

### 핑 플러드 (Ping Flood)

**특정 사이트에 많은 ICMP Echo**를 보내면, 이에 대한 응답을 하기 위해 시스템 자원을 모두 사용해버려 시스템이 정상적으로 동작하지 못하도록 하는 방법

### 랜드 어택 (Land Attack)

**출발지 IP와 목적지 IP를 같은 패킷 주소로 만들어 보냄**으로써, 수신자가 자기 자신에게 응답을 보내게 하여 시스템의 가용성을 침해

### 티어 드롭

IP 패킷의 재조합 과정에서 잘못된 Fragment Offset 정보로 인해, 수신 시스템이 문제를 발생하도록 만드는 DoS 공격

## ✏️ 그럼 어떻게 DoS 공격을 막을 수 있는가?

| 방법 | 설명 |
| --- | --- |
| 방화벽 사용 | 네트워크 트래픽을 필터링하여 악의적인 트래픽을 차단하고, 허용된 트래픽만 통과 시킨다. |
| 로드 밸런싱 | 로드 밸런서를 통해 서버의 트래픽을 분산시켜 서버에 과부하가 걸리는 것을 방지할 수 있다.  |
| ISP와 협력 | 인터넷 서비스 제공자(ISP)가 네트워크 트래픽을 모니터링하고 위협을 감지할 수 있어 ISP와 협력하여 공격을 빠르게 감지하고 대응할 수 있다. |
| DoS 공격 방지 시스템(DPS) 사용 | DPS가 네트워크 트래픽을 분석하여 공격 패턴을 감지하고 차단하는 방법이다. DPS는 빠른 시간 내에 대량의 트래픽을 처리할 수 있어 DoS 공격에 효과적으로 대응할 수 있다. |

### ✏️ 자바 개발자가 직접 DoS 공격을 처리해야 하는 것인가..? 어떻게?

자바 개발자가 직접적으로 DoS 공격을 처리하는 일은 드물지만, 서비스나 애플리케이션을 개발하고 관리하는 과정에서 DoS 공격에 대한 이해는 필요하다고 한다. 예를 들어, 서비스의 성능을 최적화하고, 과도한 요청을 관리하며, 애플리케이션의 보안을 강화하는 방법을 구현하는 데 DoS 공격에 대한 지식이 도움이 될 수 있다.

# ☻ 데이터 베이스

## 📌 분산 데이터 베이스

논리적으로는 하나의 시스템에 속하지만, 물리적으로 네트워크를 통해 여러 개의 컴퓨터 사이트에 분산되어 있는 데이터  베이스이다.

### 구성 요소로는

분산 처리기, 분산 데이터 베이스, 통신 네트워크가 있다.

- **분산 처리기(Distributed Processor)**: 여러 개의 컴퓨터로 구성되어 있으며, 각 컴퓨터는 독립적인 프로세싱 능력을 가지고 있다.
- **분산 데이터베이스(Distributed Database)**: 데이터가 여러 컴퓨터에 분산되어 저장되어 있다. 각 컴퓨터는 자신의 데이터에 대해서만 접근 권한을 가지고 있으며, 이는 네트워크를 통해 서로 공유된다.
- **통신 네트워크(Communication Network)**: 분산 처리기와 분산 데이터베이스 사이의 통신을 가능하게 하는 요소이다.

### ✏️ 분산 데이터 베이스 다시  정리

분산 데이터 베이스는 논리적으론 하나의 시스템에 속하지만, 물리적으론 네트워크를 통해 여러 컴퓨터의 사이트에 분산되어 있는 데이터 베이스이다.

> 예를 들어, 한 회사가 서울, 부산, 대구에 각각 사무실을 두고 있다고 가정해보면 각 사무실은 자신의 컴퓨터에 데이터를 저장하고 있을 것이다. 이 때, 각 사무실의 데이터를 각각 따로 조회하고 사용하는 것이 아니라, **모든 사무실의 데이터를 마치 하나의 데이터베이스에서 조회하고 사용하는 것처럼** 만드는 것이 분산 베이스이다.
>

### 분산 데이터 베이스의 목표

위치, 중복, 장애, 병행 투명성을 제공하는 것이다.

| 투명성 유형 | 설명 |
| --- | --- |
| 위치 투명성 | 실제 위치를 알 필요 없이, 논리명 만으로 액세스할 수 있다. |
| 중복 투명성 | 동일 데이터가 여러 곳에 중복되어 있더라도, 마치 하나의 데이터만 존재하는 것 처럼 사용할 수 있다. |
| 장애 투명성 | 컴퓨터, 네트워크 장애에도 불구하고 트랜잭션을 정확히 처리할 수 있다.  |
| 병행 투명성 | 다수의 트랜잭션들이 동시에 실현되더라도, 그 트랜잭션의 결과는 영향을 받지 않는다. |
