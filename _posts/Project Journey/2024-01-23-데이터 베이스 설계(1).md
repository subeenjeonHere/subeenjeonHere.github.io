# 데이터 베이스 설계 시작하기

야구 티켓 예매 사이트를 주제로 한 팀 프로젝트에서 **데이터 베이스 설계**를 담당하게 되었다.

![](https://velog.velcdn.com/images/nasubeeni/post/5e9bf093-59f3-485a-b138-aa825fb1ccde/image.png)

팀원들과 협업 도구인 Jira를 활용하여 각 기능에 대한 요구사항을 정의하였다. 요구사항을 더욱 명확히 정리하기 위해 요구사항 정의서를 별도로 문서화하였다. 최종적으로 수정된 요구사항 정의서는 아래에 기술되어 있다.

**주제:** 야구 티켓 예매 사이트

|   - 1.1. 기본 정보                    |
|     - 1.1.1. 게시글 상세 페이지에는 작성자의 닉네임이 표시되어야 한다. |
|     - 1.1.2. 게시글의 작성일자가 표시되어야 한다. |
|     - 1.1.3. 목록으로 돌아가기 버튼이 제공되어야 한다. |
|   - 1.2. 게시글 내용 표시             |
|     - 1.2.1. 게시글의 제목과 내용이 상세하게 표시되어야 한다. |
| 2. **게시판 CRUD**                    |
|   - 2.1. 작성                         |
|     - 2.1.1. 사용자는 게시글을 작성할 수 있어야 한다. |
|     - 2.1.2. 게시글에는 제목, 내용, 작성자 등의 필수 정보가 포함되어야 한다. |
|   - 2.2. 삭제                         |
|     - 2.2.1. 게시글 작성자는 자신의 글을 삭제할 수 있어야 한다. |
|     - 2.2.2. 삭제 시에는 확인 메시지를 통해 실수를 방지할 수 있는 기능이 있어야 한다. |
|   - 2.3. 수정                         |
|     - 2.3.1. 게시글 작성자는 자신의 글을 수정할 수 있어야 한다. |
|     - 2.3.2. 수정 시에는 제목과 내용을 변경할 수 있어야 한다. |
|   - 2.4. 조회                         |
|     - 2.4.1. 사용자는 게시판에서 게시된 글을 목록으로 확인할 수 있어야 한다. |
|     - 2.4.2. 게시글 목록은 최신 글이 상단에 오도록 정렬되어야 한다. |
|     - 2.4.3. 사용자는 특정 게시글을 선택하여 상세 내용을 확인할 수 있어야 한다. |
| 3. **게시글 댓글**                   |
|   - 3.1. 댓글                         |
|     - 3.1.1. 사용자는 댓글을 작성할 수 있어야 한다. |
|     - 3.1.2. 댓글은 작성자, 작성일자, 내용으로 구성되어야 한다. |
|   - 3.2. 대댓글                       |
|     - 3.2.1. 사용자는 댓글에 대댓글을 작성할 수 있어야 한다. |
|     - 3.2.2. 댓글과 대댓글은 계층적으로 표시되어야 한다. |
| 4. **알림**                           |
|   - 4.1. 커뮤니티 알림               |
|     - 4.1.1. 글에 댓글이 달리거나, 댓글에 대댓글이 달릴 경우 사용자에게 알림을 전송한다. |
|     - 4.1.2. 사용자는 받은 알림 목록을 확인하고, 알림을 클릭하면 해당 게시글 또는 댓글로 이동할 수 있다. |

> 어떻게 설계할 것인가?

데이터베이스 설계는 이전에 개인 프로젝트를 진행하면서 간단한 테이블 설계 경험은 있었지만, 이번 팀 프로젝트에서는 상대적으로 규모가 큰 테이블들을 설계해야 했다.

팀원 전체의 요구사항을 충족시키는 설계를 만들어야 했기에 조금 부담이 됐다. 그래서 더욱 꼼꼼하게 설계하고자 열심히 공부했었다.



> 어떤 순서로 설계를 할 것인가?

| 단계 | 내용 |
| --- | --- |
| 1. 요구사항 분석 | 프로젝트의 요구사항을 분석하고, 이를 바탕으로 필요한 엔터티와 속성을 식별한다. |
| 2. 엔터티 간의 관계 정의 | 다양한 엔터티 (예: 회원, 커뮤니티 게시글, 티켓 예약 등) 간의 관계를 정의한다. |
| 3. 테이블 간의 관계 설정 및 스키마 설계 | 외래키(FK)를 추가하여 관계를 명시한다. 그리고 데이터 베이스에 필요한 테이블과 스키마를 설계한다. |
| 4. 데이터 타입 및 제약조건 정의 | 각 컬럼의 데이터 타입과 제약 조건을 정의한다. 예를 들어, 비밀번호 컬럼에는 암호화된 문자열이 들어갈 수 있도록 VARCHAR 타입으로 설정하는 등, 프로젝트 구현 기능에 맞추어 데이터 타입을 설정한다. |
| 5. 인덱스 및 성능 최적화 | 데이터 베이스에 대규모 데이터가 저장될 것으로 예상되면 인덱스를 추가한다. |
| 6. 테스트 및 문서화 | 설계한 데이터 베이스가 예상한대로 기능하는지 등 테스트한다, 또한 설계 과정과 결과를 문서화 한다. |

### 엔터티 간의 관계 정의

| 엔터티 관계 | 설명 |
| --- | --- |
| 회원(Member) - 이용약관 테이블(TermsAndCondition) | 다대다(N:N) 관계. 회원은 여러 약관에 동의하고, 약관은 여러 회원에게 적용. 중개 테이블(MemberTerm)로 다대다 관계 해소. |
| 커뮤니티 게시글(CommunityPost) - 커뮤니티 댓글(CommunityComment) | 일대다(1:N) 관계. 게시글에는 여러 댓글이 달릴 수 있고, 댓글은 하나의 게시글에 속함. 외래키(FK)로 게시글과 댓글 연결. |
| 커뮤니티 게시글(CommunityPost) - 커뮤니티 알림(Notification) | 일대다(1:N) 관계. 게시글에는 여러 알림이 발생하고, 알림은 하나의 게시글에 속함. 외래키(FK)로 게시글과 알림 연결. |
| 구단커뮤니티 게시글(TeamCommunityPost) - 구단커뮤니티 댓글(TeamCommunityComment) | 일대다(1:N) 관계. 구단 게시글에는 여러 댓글이 달릴 수 있고, 댓글은 하나의 게시글에 속함. 외래키(FK)로 구단 게시글과 댓글 연결. |
| 티켓(Ticket) - 티켓 예약(Reservation) - 티켓 예약 상세(ReservationDetail) | 티켓과 예약은 일대다(1:N) 관계, 예약과 예약 상세는 일대일(1:1) 관계. 외래키(FK)로 티켓과 예약, 예약과 예약 상세를 연결. |
| 티켓 오픈 공지(TicketOpenNotice) - 티켓 오픈 공지 상세(TicketOpenNoticeDetail) | 일대일(1:1) 관계. 외래키(FK)로 티켓 오픈 공지와 상세 정보를 연결. |
| 좌석(Seat) - 경기장(Stadium) | 다대일(N:1) 관계. 여러 좌석이 하나의 경기장에 속하고, 경기장에 여러 좌석이 존재. 외래키(FK)로 좌석과 경기장을 연결. |
| 응원구단(FanTeam) - 구단(Team) | 일대일(1:1) 관계. 응원 구단은 하나의 구단에 속하고, 그 반대도 성립. 외래키(FK)로 응원 구단과 구단을 연결. |
