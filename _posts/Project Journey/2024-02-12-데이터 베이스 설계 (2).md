## 회원, 구단, 이용약관 테이블 설계

1차로 설계한 테이블을 바탕으로 팀원들과 회의하며 최종적 요구, 수정 사항까지 수령했고 이제 최종 설계할 일만 남았다.

뭔가 만족스럽지 않거나, 찝찝하면 끝날 때 까지 자꾸 파고드는 내가 데이터 베이스 설계를 맡았다. 컬럼 하나하나 신경쓰느라 당최 끝이 보이지 않는다.

그래도 처음보단 DB 설계에 조금 익숙해진 것 같다.

---

### ◼︎ 연관있는 테이블들을 일단 분류해보기

| 테이블 종류                    | 설명                                               |
|---------------------------|--------------------------------------------------|
| 회원 테이블, 이용약관, 이용약관 중개 테이블 | 회원 정보와 이용약관 데이터를 저장하고, 회원 ↔ 이용약관 간의 동의 상태를 연결한다. |
| 일반 커뮤니티 게시글, 댓글, 알림 테이블   | 커뮤니티 내의 게시글, 댓글 그리고 알림 데이터를 관리한다.                |
| 구단 별 커뮤니티 게시글, 댓글, 알림 테이블 | 각 구단 별로 커뮤니티 게시글, 댓글 그리고 알림 데이터를 관리한다.           |
| 게임 - 티켓 - 티켓 예약 - 경기장 테이블 | 티켓 예매 기능 구현을 위한 데이터 테이블이다.                       |
| 티켓 오픈 공지, 오픈 공지 상세 테이블    | 티켓 오픈에 관한 공지와 그 공지의 세부 사항을 저장한다.                 |

# ◎ 회원 테이블

야구 예매 커뮤니티 사이트에서 회원 테이블은 사이트에서 각 사용자를 고유하게 식별하고, 해당 사용자의 정보를 저장하는 역할을 한다. 이 테이블에는 사용자의 아이디, 비밀번호, 닉네임, 주민번호, 응원하는 구단의
ID, 핸드폰 번호, 이메일, 그리고 권한 여부 등의 정보가 포함된다.

| 회원     |       |                    |    |    |             |            |                   |
|--------|-------|--------------------|----|----|-------------|------------|-------------------|
| member | 회원id  | mbr_seq            | PK |    | INT         | AI,NOTNULL |                   |
| member | 아이디   | mbr_id             |    |    | VARCHAR(20) | NOTNULL    |                   |
| member | 비밀번호  | mbr_pwd            |    |    | VARCHAR(60) | NOTNULL    |                   |
| member | 닉네임   | mbr_nm             |    |    | VARCHAR(20) | NOTNULL    |                   |
| member | 주민번호  | mbr_residentnumber |    |    | VARCHAR(14) | NOTNULL    |                   |
| member | 구단id  | team_id            |    | FK | INT         | NOTNULL    | DEFAULT 값 0 기본 구단 |
| member | 핸드폰번호 | mbr_phone          |    |    | VARCHAR(15) | NOTNULL    |                   |
| member | 이메일   | mbr_email          |    |    | VARCHAR(50) | NOTNULL    |                   |
| member | 권한여부  | mbr_role           |    |    | INT         | NOTNULL    | 사용자의 권한 상태를 판별    |

### ◼︎ 참조키에 대한 설명

**구단id (**team_id**):**

응원하는 구단의 아이디를 나타내는 외래키(Foreign Key)이다. 구단 테이블의 구단 id를 참조하며, 회원이 응원하는 구단을 선택하지 않으면 기본값 0이 설정된다.

### ◼︎ 제약 조건에 대한 설명

- NOT NULL 제약조건은 해당 필드가 반드시 값을 가져야함을 의미한다. 회원 테이블에서 주민번호, 비밀번호, 아이디 등의 값들에 NOT NULL 제약조건을 설정해주었다.
    - 회원가입 시 필수적으로 기입해야 하는 정보이다. 이들 중 어느 하나라도 누락된다면, 회원 식별 및 서비스 제공에 문제가 발생할 수 있기 때문이다.
- 권한여부 (mbr_role)
    - 팀원의 요구사항을 반영하여 사용자의 권한 상태를 판별하도록 했다. 관리자 vs 일반 회원 권한을 부여하기 위해 필요한 속성이다.

### ◼︎ 수정사항

- **권한여부를 나타내는 컬럼 명**
    - `is_grant`, `is_normal`
        - 회원이 일반회원인지 관리자 회원인지를 구별하기 위한 컬럼이다. 컬럼에서 부정의 의미는 내포하지 않고자 했다. 관리자 회원은 소수이고, 일반회원이 대부분이기에 is_normal로 설정하였다. 또
          관리자 테이블을 따로 생성할 수 있지만, 관리자로서의 권한은 추후 프로젝트 시 확장 예정이기에 우선 컬럼으로 판별하도록 했다.
        - 1월 10일 수정: mbr_role로 수정하였다. Spring Security를 도입하면서 회원의 권한을 저장하기 위해.
- **관리자 테이블을 별도로 생성해야 했는가?**
    - 회원과 관리자의 정보가 확연하게 다르고, 두 그룹간에 공통 정보가 없다면 테이블을 나누어주는 것이 적절할 것이다. 다만 이번 프로젝트의 규모나 최종 요구사항 정의서를 고려했을 때 두 테이블을 분리할 필요는
      없다고 생각했고, 회원 테이블 내에서 구분해도 될 것이라 생각했다.

# ◎ 구단 테이블

구단 테이블은 야구 예매 사이트에서 각 야구 팀을 고유하게 식별하고, 해당 팀의 정보를 저장하는 역할을 한다. 이 테이블에는 구단의 고유 ID와 이름 등의 정보가 포함된다. 또한, 이 테이블은 다른 테이블과의 관계를
통해 응원 구단, 경기장, 경기 등과 연결되어 각 기능을 수행하도록 한다.

| 구단   |      |           |    |              |
|------|------|-----------|----|--------------|
| team | 구단id | team_id   | PK | INT          |
| team | 구단명  | team_name |    | VARCHAR(50)  |
| team | 구단로고 | team_logo |    | VARCHAR(255) |

### ◼︎ 컬럼 설명

- **구단id**
    - 각 구단을 고유하게 식별할 기본키 (Primary Key)이다. 자동 증가하는 정수형 값으로, 10개의 KBO리그 구단을 저장할 것이다.
    - 또한, 회원이 응원구단을 선택하지 않을 경우를 대비하여 DEFAULT 값 “0” 기본 구단을 선택한 것으로 한다. 즉, 10개의 구단 데이터와 1개의 기본 구단 데이터로 총 11개의 데이터가 생성될
      것이다.

### ◼︎ 테이블 예시

```jsx
team_id | team_name | team_logo
-------- | -------------------- | ------------------------
    1 | 삼성
라이온즈 | /logos/s
amsung_logo.png
2 | 한화
이글스 | /logos/
hanwha_logo.png
3 | 롯데
자이언츠 | /logos/
lotte_logo.png
...     | ...                | ...
```

# ◎ 이용약관, 이용약관 중개 테이블

### ◼︎ 이용약관 테이블

| 이용약관           |      |              |    |      |
|----------------|------|--------------|----|------|
| term_condition | 약관id | term_id      | PK | INT  |
| term_condition | 약관코드 | term_code    |    | INT  |
| term_condition | 약관내용 | term_content |    | TEXT |

<br>



회원 테이블은 각 **회원을 고유하게 식별**하며, 이용약관 테이블은 **다양한 이용약관을 저장**한다. 따라서 회원과 이용약관 테이블은 1:N 관계이다.

<br>

### ◼︎ 컬럼 설명

- 약관id (term_id)
    - 약관의 고유 식별자로 사용되는 기본키(PK)이다.
- 약관 코드(term_code)
    - 약관을 구분하는 데 사용되는 코드이다.
- 약관 내용(term_content)
    - 약관의 실제 내용을 나타낸다. TEXT 형식으로 정의되어 있다.
    - TEXT 데이터 타입은 가변 길이의 문자열을 저장할 수 있다. 매우 큰 길이의 문자열도 저장할 수 있으므로, 긴 텍스트 내용이나 HTML 코드 등을 저장할 때 주로 사용된다.

<br>

### ◼︎ 이용약관중개 테이블

| 회원이용약관중개         | 논리명  | 물리명      |    |    |        |         |
|------------------|------|----------|----|----|--------|---------|
| member_termagree | 중개id | mt_id    | PK |    |        | INT     |
| member_termagree | 회원id | mbr_id   |    | FK | UNIQUE | INT     |
| member_termagree | 약관id | term_id  |    | FK | UNIQUE | INT     |
| member_termagree | 동의여부 | is_agree |    |    |        | BOOLEAN |

회원은 **여러 개의 이용약관을 선택(동의)**할 수 있으며**(1:N 관계),** 반대로 한 개의 이용약관은 **여러 회원에 의해 선택될 수 있다(N:M 관계).** 이러한 관계를 표현하기 위해 이용약관 중개
테이블이 사용되며, 이 테이블은 회원의 ID와 이용약관 코드를 외래 키로 사용하여 회원과 이용약관 사이의 관계를 나타낸다.

<br>

### ◼︎ 컬럼 설명

- 중개id (mt_id)
    - 중개의 고유 식별자로 사용되는 기본 키(PK)이다.
- 회원id (mbr_id)
    - 회원 테이블의 회원id(member_id)를 참조한다. 외래 키(FK)로 설정되어 있으며, 회원을 식별하는 데 사용된다.
- 약관id (term_id)
    - 약관 테이블의 약관id(term_id)를 참조한다. 외래 키(FK)로 설정되어 있으며, 약관을 식별하는 데 사용된다.
- 약관코드(term_code)
    - 중개 테이블과 이용약관 테이블간의 약관 코드를 참조한다.
- 동의여부(is_agree)
    - 회원이 이용약관에 동의했는지의 여부를 나타낸다. BOOLEAN 데이터 형식으로 동의했을 경우 TRUE, 동의하지 않았을 경우 FALSE로 정의되어 있다.

      <br>

### ◼︎ 제약조건

- **UNIQUE(회원id, 약관id)**
    - 데이터 무결성을 보장하기 위해, 회원id와 약관 코드의 조합의 UNIQUE 제약조건을 생성했다.
    - 회원 A, 이용약관코드 A,B,C가 있다면 회원은 A,B,C 각 하나의 약관코드에 동의/비동의 할 수 있기 때문이다.

<br>

### ◼︎ 고려사항

1. 회원이 여러 개의 이용 약관을 선택할 수 있다.
2. 약관 코드와 약관 내용이 함께 저장되어야 한다.
3. 하나의 이용 약관이 여러 회원에게 선택되어야 한다.

즉, 회원 ↔ 약관간의 다대다(N:M) 관계를 나타내는 **중개 테이블**을 생성해야 한다. 또한, 정규화는 제 3정규화까지 수행하도록 했다.

<br>

#### ◼︎ 중개 테이블이 필요한 이유

명확한 데이터 구조화를 위해서다. 이용약관은 회원가입 시에만 사용되는 것이 아니라, 사이트의 Footer에서도 이용약관 정보를 제공하고자 하는 요구사항이 있었다. 이로인해 이용약관 자체의 데이터를 가지는 테이블이
필요했다.

이용약관 내용 자체를 독립적으로 저장하는 테이블을 가짐으로써, 추후 이용약관의 내용이 변경되거나 새로운 약관이 추가될 때 기존 데이터 구조에 영향을 주지않고 관리할 수 있을 것이다.

## **◼︎ 테이블 예시:**

### 이용약관 테이블 (term_condition):

```sql
term_code
| content
---------|---------------------------------------------
1        | 이용약관 내용1...
2        | 이용약관 내용2...
3        | 이용약관 내용3...
```

### 회원이용약관 중개 테이블 (member_termagree):

```sql
member_id
| term_code | isagree
----------|----------|--------
1         | 1        | true
1         | 2        | true
2         | 1        | false
3         | 3        | true
```

이용약관중개 테이블, 이용약관 테이블의 term_code가 외래키 관계를 가지도록 한다.

<br>

### 정규화

**1. 1NF (First Normal Form)**

```sql
이용약관
테이블 (term_condition)
+------------------------+
| 약관코드 (term_code)   | 약관내용 (term_content) |
+------------------------+
| 001                  | 이용약관 A              |
| 002                  | 이용약관 B              |
| 003                  | 이용약관 C              |
+------------------------+
```

각 행은 고유한 약관코드를 가지고 있으며, 약관 내용이 해당 약관코드에 종속되도록 한다.

**2. 2NF (Second Normal Form)**

```sql
회원이용약관중개
테이블 (member_termagree)
+---------+------------+----------+
| member_id | term_code | is_agree |
+-----------+-----------+---------+
|     101   |    001    |   true  |
|     101   |    002    |  false  |
|     102   |    001    |   true  |
|   ...     |    ...    |   ...   |
```

중개 테이블의 `PK`는 `member_id`와 `term_code`이며, `is_agree`는 `term_code`에만 종속되어 있으므로 2NF를 만족한다.

---

### 다대다(N:M) 관계 추가 정리

다대다 관계는 여러 개의 레코드가, 다른 테이블의 여러 개의 레코드와 관계있는 것이다.

1. 우선 여러개의 이용 약관과, 여러 명의 회원이 있다.
2. 한 명의 회원은 여러 개의 이용 약관을 선택할 수 있고, 이용약관 하나는 여러 명의 회원에게 선택될 수 있으므로 N:M 관계이다.
3. 이 경우, 두 개의 테이블과 1:N 관계를 나타낼 새로운 테이블로 N:M 관계를 나타낼 수 있다.
4. 즉, 중개 테이블은 회원과 이용 약관 사이에서 1:N 관계로 묶어주는 역할을 한다.

---

### 회의 이후 업데이트

1. 프로필 사진 컬럼 삭제
    2. 회원의 프로필 사진을 등록을 위해 프로필 사진 컬럼이 있었으나 이를 삭제했다. 대신 응원구단을 선택하면 해당 구단의 로고를 사용한다. 또한 구단을 선택하지 않은 회원에게는 기본 프로필 사진이 제공된다.
